## 1. 背景

当前右侧栏已经有“大纲”标签，但仍处于占位状态，无法真正帮助用户在长文中快速定位章节。与此同时，编辑区与预览区已经具备按行号同步定位的基础设施，标题解析与锚点查找逻辑也已经分散存在于前后端多个位置。

这意味着问题已经不在于“能不能跳转”，而在于缺少一个统一的大纲能力，把当前笔记里的标题结构稳定提取出来，并接入现有双栏同步导航链路。

对知识库场景来说，大纲的价值不是装饰性目录，而是：

1. 帮助用户在长笔记中快速理解结构。
2. 让用户一跳直达目标章节，而不是手动滚动查找。
3. 让编辑区与预览区始终落在同一章节，减少定位成本。

## 2. 目标

本期目标如下：

1. 大纲只作用于当前打开的单篇笔记，不扩展到整库或跨笔记目录。
2. 只收录当前笔记中的 Markdown 标题，层级范围为 `#`、`##`、`###`。
3. 右侧栏展示稳定、可点击的大纲结构，不做首版折叠树复杂交互。
4. 点击任意大纲项后，编辑区与预览区必须同步跳转到同一章节。
5. 大纲生成规则应尽量复用现有 Markdown 解析与导航能力，避免前后端规则分叉。

## 3. 非目标

本轮明确不做以下内容：

- 不做整库目录、跨笔记目录或按文件夹聚合的大纲。
- 不做四级及以下标题展示。
- 不做大纲拖拽排序、折叠持久化、章节重命名等编辑型能力。
- 不做根据当前滚动位置自动激活当前章节的复杂联动，若实现成本低可后续追加，但不作为首版交付门槛。
- 不把 front matter 的 `title` 或笔记文件名虚拟成目录节点；首版只认正文中的 Markdown 标题。

## 4. 方案对比与结论

### 方案 A：前端直接解析当前正文

做法：前端在右侧栏读取当前编辑内容，自行解析标题层级、组织树并完成跳转。

优点：

- 前端改动集中。
- 首次看起来实现路径较短。

缺点：

- 会复制一套标题解析规则，容易与后端现有逻辑漂移。
- 后续如果搜索、链接、目录都要识别相同标题规范，维护成本会持续上升。
- front matter、setext 标题、锚点归一化等边界容易出现前后不一致。

### 方案 B：后端提供单篇笔记大纲接口，前端只负责展示与导航

做法：后端解析当前笔记，返回结构化大纲数据；前端右侧栏渲染并在点击时复用现有导航状态。

优点：

- 标题解析规则集中，便于复用与测试。
- 与现有按行号导航链路天然兼容。
- 后续扩展到内部链接、章节搜索、整库目录时可继续复用。

缺点：

- 首版需要补一个 Tauri 命令和对应测试。
- 数据流会比纯前端方案多一层请求。

### 方案 C：基于预览 DOM 反向抓取标题

做法：只在预览渲染后，从 DOM 中读取标题节点构造大纲。

优点：

- 视觉结果直观。

缺点：

- 依赖预览面板已渲染，模式切换和内容刷新时机不稳定。
- 对源码行号映射不天然可靠。
- 与“编辑区和预览区同步跳转”目标不匹配。

### 结论

本期采用方案 B。

原因：它能最大程度复用现有 Markdown 解析与搜索导航基础设施，把“标题如何识别、落在哪几行”作为后端统一建模结果输出，前端只承担展示与交互职责，边界最清晰。

## 5. 产品定义

本期将“大纲”定义为：

- 当前打开笔记的章节结构视图。
- 单篇笔记内部的快速导航入口。
- 编辑区与预览区的共用章节定位器。

用户在右侧栏看到的每一项不表示“一个可编辑章节对象”，而表示“当前笔记中的一个可跳转标题块”。

因此，大纲项至少要具备以下语义：

- 显示文本是什么。
- 它属于哪一级标题。
- 它在源码里起止于哪些行。
- 点击后应该把双栏定位到哪里。

## 6. 信息模型

建议的大纲项模型如下：

```ts
export interface NoteOutlineItem {
  id: string;
  text: string;
  level: 1 | 2 | 3;
  lineStart: number;
  lineEnd: number;
  anchor: string;
  children: NoteOutlineItem[];
}
```

字段约束如下：

### 6.1 `id`

- 用于前端渲染 key 和选中态识别。
- 必须稳定且能区分同名标题。
- 推荐使用 `anchor + ":" + lineStart` 生成。

### 6.2 `text`

- 为标题的纯文本内容。
- 不保留 Markdown `#` 前缀。
- 不保留链接、强调等语法标记的装饰性符号。

### 6.3 `level`

- 仅允许 `1 | 2 | 3`。
- 超过 `###` 的标题首版直接忽略，不进入大纲。

### 6.4 `lineStart` 与 `lineEnd`

- 表示该标题块在原始 Markdown 中的起止行。
- `lineStart` 是标题所在行。
- `lineEnd` 是下一个同级或更高层级标题前一行；如果已经是文末，则取文末行号。
- 该区间直接复用为前端 `searchNavigationTarget` 的定位范围。

### 6.5 `anchor`

- 用于预留章节锚点语义。
- 首版点击跳转主通道仍以行号为准，但 `anchor` 需要随模型返回，避免后续再补数据迁移。

### 6.6 `children`

- 首版前端仍按树状数据接收。
- 即便展示时不支持折叠，也应保留层级关系，避免未来改接口。

## 7. 后端接口与解析设计

### 7.1 命令定义

后端新增单篇笔记大纲命令，建议接口形态如下：

```ts
getNoteOutline(path: string): Promise<NoteOutlineItem[]>
```

输入约束：

- `path` 必须是当前知识库内的笔记路径。
- 若文件不存在、不是 Markdown 笔记或无法解析，返回空数组或结构化错误，但不得让前端右侧栏整体崩溃。

### 7.2 解析规则

首版标题解析规则建议如下：

1. 支持 ATX 标题，即 `#` 到 `###`。
2. 支持 Setext 标题时，只在其等价层级属于 `h1` 或 `h2` 时纳入。
3. 忽略代码块中的伪标题文本。
4. 忽略 front matter 区域中的键值行。
5. 标题文本需要做与现有锚点规则一致的归一化，避免大纲点击与内部链接跳转目标不一致。

### 7.3 行号区间计算

每个标题节点需要同时生成“标题行”与“章节覆盖范围”。

推荐规则：

1. 当前标题的 `lineStart` 等于标题出现行。
2. 从当前标题之后向下查找下一个层级小于等于当前层级的标题。
3. 找到后，当前节点 `lineEnd` 取其前一行。
4. 若未找到，则 `lineEnd` 取正文最后一行。

这样做的目的是保证点击某个大纲项时，编辑器与预览区不仅能滚到标题行，还能用同一段范围做高亮和后续“当前章节识别”。

### 7.4 结构组织

后端应直接返回树状结构，而不是平铺数组。

组织规则：

1. `h1` 作为顶层。
2. `h2` 挂在最近的上一个 `h1` 下。
3. `h3` 挂在最近的上一个 `h2` 下；若不存在 `h2`，则挂到最近 `h1` 下。
4. 若文档以 `h2` 开头，则该 `h2` 可直接作为顶层节点，前端不应因为缺少 `h1` 而丢失它。

## 8. 前端状态与交互设计

### 8.1 展示位置

继续复用右侧栏现有 `outline` 标签，不新增新的顶级面板。

### 8.2 加载态

大纲面板需要至少处理三种状态：

1. 无当前笔记：提示“打开笔记后显示大纲”。
2. 当前笔记无标题：提示“当前笔记暂无可用标题”。
3. 大纲已生成：展示目录列表。

首版不要求复杂骨架屏，但应避免空白面板。

### 8.3 列表展示

展示规则：

1. 按文档顺序显示。
2. 使用缩进表达 `1/2/3` 级层级关系。
3. 当前首版不做展开/收起控件。
4. 标题过长时单行截断，但 hover 需能看到完整文本或 title 提示。

### 8.4 点击行为

点击任意大纲项时，前端统一写入现有导航状态，而不是单独发明“大纲跳转状态”。

建议写入值如下：

```ts
setSearchNavigationTarget({
  note_id: currentNote.id,
  note_path: currentNote.path,
  note_title: currentNote.title,
  line_start: item.lineStart,
  line_end: item.lineEnd,
  occurrence_order: 1,
  match_text: item.text,
  context_snippet: item.text,
  source: "body",
  revision: Date.now(),
});
```

这样处理的好处是：

- 编辑器会沿用现有按行号滚动与高亮逻辑。
- 预览区会沿用现有 source line 同步滚动与高亮逻辑。
- 大纲不需要引入新的跨组件通信机制。

### 8.5 双栏同步要求

用户已经明确选择“编辑区与预览区都同步跳转”，因此这是首版硬要求：

1. 点击后编辑区必须滚到目标标题行附近。
2. 点击后预览区必须滚到对应标题块。
3. 两侧应保持同一章节语义，而不是一边落到标题、一边落到正文中部。

### 8.6 当前章节高亮

首版基础要求是“点击哪个亮哪个”。

如果现有状态链路已经足以低成本判断当前目标，可在点击后让该大纲项进入选中态；但不要求首版实现基于手动滚动的实时跟随高亮。

## 9. 刷新与同步策略

### 9.1 触发时机

大纲刷新建议由以下事件触发：

1. 当前笔记切换时立即刷新。
2. 当前笔记内容变化时做防抖刷新。
3. 当前笔记保存成功后允许再次校正，但不要求额外强刷。

### 9.2 防抖策略

建议前端对“内容变化触发大纲刷新”增加 200 到 300ms 防抖。

原因：

- 输入标题时用户会连续敲击，逐键请求没有必要。
- 大纲不是逐字符实时预览型能力，允许轻微延迟。
- 这样能显著降低 Tauri 往返频率。

### 9.3 失败处理

若大纲请求失败：

- 面板内显示轻量错误提示。
- 不影响编辑、保存、预览等主链路。
- 再次编辑或切换笔记时允许自动重试。

## 10. 测试与验收

### 10.1 后端测试

至少覆盖以下场景：

1. 只有 `#` 标题的简单文档。
2. `#`、`##`、`###` 混合嵌套文档。
3. 以 `##` 开头、没有 `#` 顶层的文档。
4. 代码块中包含 `# 标题` 文本但不应进入大纲。
5. Setext 一级/二级标题能被正确识别。
6. 重名标题能生成不同 `id`。
7. `lineEnd` 计算正确，不会跨到下一个同级章节之后。

### 10.2 前端测试

至少覆盖以下场景：

1. 无当前笔记时显示空态。
2. 无标题笔记时显示“暂无可用标题”。
3. 有大纲时按层级正确渲染。
4. 点击大纲项后写入正确的 `searchNavigationTarget`。
5. 点击后编辑区与预览区都收到一致的导航目标。
6. 内容快速变化时，大纲请求经过防抖，不会无上限触发。

### 10.3 人工验收

人工验收至少验证：

1. 打开长笔记后，右侧栏能稳定显示到 `###` 的目录。
2. 点击任意目录项，编辑区与预览区都能跳到正确章节。
3. 新增或删除标题后，大纲会在短延迟后更新。
4. 含 front matter、代码块和中文标题的文档，目录结果仍然正确。

## 11. 风险与回退

### 11.1 规则漂移风险

如果前端再次私自实现一套标题解析，后续会与搜索、链接、预览的标题语义逐渐偏离。因此本期必须坚持“后端建模，前端消费”。

### 11.2 性能风险

若对每次输入都实时请求大纲，长文下可能引发频繁解析。通过前端防抖和复用现有解析能力可以把风险控制在可接受范围内。

### 11.3 可回退策略

若首版联调中发现树状展示或自动刷新复杂度超预期，可保留后端接口不变，临时把前端降级为：

- 平铺展示大纲项。
- 仅在切换笔记时刷新。

但不能回退到“只显示占位文本”。因为一旦进入实现阶段，最小可交付也必须包含真实目录和真实跳转。
