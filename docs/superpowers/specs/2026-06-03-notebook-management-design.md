## 1. 背景

MyNote 当前已经支持创建笔记本，并把笔记本视觉属性持久化到知识库级元数据文件。当前模型中，只有 `notes/<top-level-dir>` 被视为笔记本，根目录下的独立笔记以“未归档”形式展示。

这套模型已经能支持：

- 创建笔记本
- 创建笔记时选择目标笔记本
- 在侧栏展示顶级笔记本的图标和颜色
- 将视觉属性持久化到知识库级元数据文件

但当前笔记本仍缺失完整管理能力：

- 不能重命名笔记本
- 不能修改已创建笔记本的图标和颜色
- 不能删除空笔记本
- 不能调整多个笔记本在侧栏中的顺序

本设计在不改变现有笔记本语义的前提下，补齐上述管理能力。

## 2. 目标

本次设计目标：

- 支持重命名顶级笔记本目录。
- 支持修改顶级笔记本的图标和颜色。
- 支持删除空笔记本，禁止删除非空笔记本。
- 支持调整顶级笔记本顺序，并持久化排序结果。
- 保持“笔记本仅等于 `notes/<top-level-dir>`”这一模型不变。
- 保持“未归档”是 UI 合成分组，不进入笔记本管理能力。
- 在目录重命名后，保证文件系统、数据库路径、前端树和视觉元数据保持一致。

## 3. 非目标

本轮不做以下内容：

- 不支持把笔记本移动到 `notes/` 之外或嵌套到其他目录下。
- 不支持删除非空笔记本时自动迁移笔记到“未归档”。
- 不支持递归删除包含笔记或子目录的笔记本。
- 不支持拖拽排序笔记本。
- 不支持修改“未归档”的名称、顺序或视觉属性。
- 不支持批量编辑多个笔记本。
- 不改变现有文章拖拽移动逻辑。

## 4. 设计原则

- 笔记本语义继续采用路径模型，避免引入新的独立实体表。
- 视觉属性和顺序继续集中持久化到知识库级元数据文件，避免多份真源。
- 删除操作遵循保守策略，只允许删除空笔记本。
- 目录重命名必须是端到端一致更新，不能只改文件系统或只改数据库。
- 前端交互优先采用显式、低风险方案，排序采用“上移 / 下移”而不是新拖拽语义。
- 错误优先可解释，不做静默失败或隐式数据迁移。

## 5. 方案选择

### 方案 A：扩展现有知识库级笔记本元数据文件并新增笔记本命令（推荐）

在现有 `.mynote/notebook-visuals.json` 管理模型上继续扩展，使每个顶级笔记本记录同时保存图标、颜色和顺序。后端新增笔记本级命令，分别负责重命名、更新元数据、删除空笔记本和更新顺序。

优点：

- 图标、颜色、顺序在同一份真源中维护。
- 与现有笔记本路径模型一致，不引入额外数据库实体。
- 便于在构建树时一次性把视觉属性和顺序应用到顶级笔记本。

缺点：

- 目录重命名时需要同步更新文件系统、数据库路径和元数据键，要求实现完整。

### 方案 B：视觉属性和排序拆成两份元数据文件

继续保留当前视觉属性文件，新增独立排序文件。

优点：

- 职责分离更明显。

缺点：

- 同一笔记本信息分散在两份文件中，后续维护和故障恢复更复杂。
- 重命名时需要同步迁移两份记录，收益不高。

### 方案 C：前端仅保存临时排序

排序仅保存在前端状态或浏览器缓存中，不写入知识库。

优点：

- 实现速度快。

缺点：

- 不是真正的知识库级排序，不满足跨会话和跨设备一致性。
- 与现有视觉属性已持久化的方向不一致。

本轮采用方案 A。

## 6. 详细设计

### 6.1 能力边界

本轮新增四项笔记本管理能力：

- 重命名笔记本
- 编辑笔记本图标和颜色
- 删除空笔记本
- 上移或下移笔记本顺序

适用对象仅限顶级笔记本，也就是满足以下条件的目录节点：

- `node.is_dir = true`
- `path` 形如 `notes/<name>`
- `path` 不是 `notes/__unarchived__`

### 6.2 重命名笔记本

重命名是本轮最核心的行为。

操作输入：

- `old_path`: 现有笔记本路径，例如 `notes/法律`
- `new_name`: 新笔记本名称，例如 `案例库`

后端行为：

1. 校验 `old_path` 必须是顶级笔记本路径。
2. 校验 `new_name` 非空、不是保留路径段、不包含路径分隔符。
3. 将新名称规范化为安全目录名。
4. 如果目标目录已存在，则拒绝。
5. 执行目录重命名：`notes/法律` -> `notes/案例库`。
6. 批量更新该目录下所有笔记的数据库 `path` 字段。
7. 对受影响的笔记重新建立索引或同步更新索引相关路径数据。
8. 将知识库级笔记本元数据记录从旧路径键迁移到新路径键，并保留图标、颜色和顺序。
9. 返回重命名结果，包括新笔记本路径以及受影响的当前笔记路径映射信息。

前端行为：

- 提交成功后刷新树。
- 如果当前打开笔记位于被重命名的笔记本内，需要同步编辑器当前笔记路径和侧栏选中路径。
- 失败时保留输入草稿并展示错误。

### 6.3 编辑图标和颜色

此能力仅修改知识库级元数据文件，不涉及文件系统或数据库路径。

操作输入：

- `notebook_path`
- `icon`
- `color`

后端行为：

- 校验目标路径是顶级笔记本。
- 读取现有元数据文件。
- 更新对应记录的图标和颜色，保留顺序和未知字段。
- 写回元数据文件。

前端行为：

- 使用轻量编辑面板复用现有新建笔记本的图标和颜色预设。
- 保存成功后刷新树并关闭面板。
- 保存失败时保留当前选择。

### 6.4 删除空笔记本

删除采用保守策略：只有空笔记本才能删除。

空笔记本定义：

- 对应目录下没有 Markdown 笔记
- 对应目录下没有任何子目录
- 对应目录本身存在且是顶级笔记本目录

后端行为：

1. 校验路径是顶级笔记本。
2. 读取目录内容。
3. 只要存在文件或子目录，就返回“非空笔记本不可删除”。
4. 目录为空时删除目录。
5. 清理元数据文件中对应记录。
6. 返回删除成功。

前端行为：

- 删除前弹出确认。
- 删除成功后刷新树。
- 若删除目标恰好是当前选中的目录，则清空对应选中状态。
- 删除失败时显示明确原因，不做本地移除。

### 6.5 调整笔记本顺序

排序范围只覆盖顶级笔记本，不包括“未归档”。

本轮交互采用显式操作：

- 上移
- 下移

后端行为：

- 接收完整的顶级笔记本顺序列表，或接收源路径加目标相邻位置并在服务端转换为完整顺序。
- 把顺序写入元数据文件中的 `order` 字段。
- 对没有顺序值的新笔记本，在读树时自动排到已有顺序之后。

树排序规则：

1. 有顺序值的顶级笔记本按 `order` 升序排列。
2. 没有顺序值的顶级笔记本排在后面。
3. 对同组内顺序冲突或缺失情况，用名称作为稳定次排序。
4. “未归档”始终固定在所有顶级笔记本之后。

### 6.6 UI 入口

每个顶级笔记本节点增加“更多操作”入口，包含：

- 重命名笔记本
- 编辑图标和颜色
- 上移
- 下移
- 删除笔记本

入口只对顶级笔记本显示，不对普通子目录、普通文章和“未归档”显示。

## 7. 后端接口与数据变更

### 7.1 元数据文件扩展

当前知识库级元数据文件需要从“只保存图标和颜色”扩展为“保存图标、颜色和顺序”。

记录示意：

```json
{
  "notes/法律": {
    "icon": "book",
    "color": "blue",
    "order": 10
  }
}
```

要求：

- 继续保留未知字段，避免覆盖未来扩展信息。
- 继续保留无关非法记录，避免用户已有文件被粗暴重写。
- 继续沿用当前进程内串行写保护策略。

### 7.2 Tauri 命令新增

建议新增以下命令：

- `rename_notebook(oldPath, newName)`
- `update_notebook_visual(notebookPath, icon, color)`
- `delete_notebook(notebookPath)`
- `reorder_notebooks(orderedPaths)`

这些命令应落在现有 note command 体系内，继续由 `src-tauri/src/commands/note.rs` 暴露，并在 `src-tauri/src/services/note.rs` 与 notebook visual service 中完成实现。

### 7.3 前端 API 扩展

`src/api/commands.ts` 需新增对应 facade。

`src/hooks/useKnowledgeBase.ts` 需新增对应 hook 能力：

- `renameNotebook`
- `updateNotebookVisual`
- `deleteNotebook`
- `reorderNotebooks`

这些 hook 在成功后统一刷新树，并按需要同步当前选中路径与当前打开笔记路径。

## 8. 前端交互设计

### 8.1 菜单与面板

前端在左侧笔记本树里为顶级笔记本提供更多操作按钮。

交互结构：

- 点击更多按钮打开菜单。
- 点击“重命名笔记本”进入重命名输入态。
- 点击“编辑图标和颜色”打开视觉属性面板。
- 点击“上移 / 下移”立即执行顺序调整。
- 点击“删除笔记本”弹出确认。

### 8.2 禁用态

下列场景需要显式禁用：

- 第一个笔记本禁用“上移”。
- 最后一个笔记本禁用“下移”。
- “未归档”没有更多操作入口。
- 普通目录和普通文章没有笔记本管理入口。

### 8.3 成功与失败反馈

成功：

- 刷新树并关闭对应面板或菜单。
- 保持当前上下文尽量稳定。

失败：

- 错误文案应直接说明原因，例如“笔记本不为空，无法删除”。
- 对重命名和视觉编辑，保留用户输入或选择，便于重试。

## 9. 错误处理与边界

需要显式处理以下边界：

- 新名称为空。
- 新名称规范化后为空。
- 新名称与已有笔记本冲突。
- 目标路径不是顶级笔记本。
- 删除目标目录不存在。
- 删除目标目录非空。
- 元数据文件损坏时，更新视觉属性或顺序应拒绝覆盖。
- 重命名目录成功后若元数据迁移或索引更新失败，需要回滚或以事务式思路保证最终一致性。
- 受影响笔记如果正处于编辑态，前端路径同步不能把当前页面切成“文件不存在”。

## 10. 测试与验证策略

### 10.1 前端测试

补充以下测试：

- 顶级笔记本显示更多操作入口。
- 普通目录、普通文章、未归档不显示该入口。
- 重命名笔记本成功后刷新并显示新名称。
- 编辑图标颜色后 marker 正确更新。
- 上移和下移后顶级笔记本顺序更新。
- 第一项和最后一项的排序按钮禁用正确。
- 删除空笔记本成功。
- 删除非空笔记本失败时展示错误且树不变。

### 10.2 后端测试

补充以下测试：

- `rename_notebook` 只允许顶级笔记本。
- 重命名后目录真实变更。
- 重命名后所有受影响笔记路径被同步更新。
- 重命名后元数据键迁移成功并保留图标、颜色、顺序。
- 删除空笔记本成功并清理元数据。
- 删除非空笔记本返回错误。
- 顺序写入后，构建树时顶级笔记本按预期排序。
- 无顺序值的新笔记本按回退规则稳定排序。

### 10.3 验证命令

实现完成后，至少执行：

- 侧栏相关 Vitest 聚焦测试
- `corepack pnpm build`
- `cd src-tauri && cargo test`

## 11. 后续扩展

本轮完成后，可作为后续增强的基础：

- 将上移 / 下移升级为笔记本拖拽排序
- 支持批量整理多个笔记本
- 支持非空笔记本删除前的迁移向导
- 支持修改笔记本描述、封面或更多视觉属性
