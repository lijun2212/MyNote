## 1. 背景

MyNote 当前已经具备或已明确规划以下能力：

- 知识库创建、打开、最近使用。
- 笔记本与笔记管理。
- Markdown 编辑、预览、自动保存。
- 标签列表、标签筛选、标签上下文导航。
- Wiki 链接、反向链接、未解析链接。
- 搜索面板与文件树过滤。
- 笔记拖拽移动与笔记本管理。
- 后续将继续建设图谱、修订记录、知识关系、导出与多窗口等能力。

当前问题不是“缺少命令入口”，而是“入口分布尚未形成稳定信息架构”。如果继续以零散按钮、内联动作和临时右键项推进，后续功能一旦增多，系统菜单和对象菜单会迅速失衡，导致：

- 同一动作在多个地方命名不一致。
- 高级能力只能塞进不稳定的弹层或临时按钮。
- 右键菜单越做越长，无法形成对象心智。

因此本设计的核心目标是先固定骨架，再逐步填充功能，而不是每加一个功能就重排一次菜单。

## 2. 目标

本轮设计目标如下：

- 定义一套稳定、可扩展的系统菜单结构。
- 定义一套按对象分型的右键菜单结构。
- 让当前已实现能力可以自然落位，不依赖临时按钮堆叠。
- 为图谱、修订、关系管理、导出等未来能力预留位置，但不让菜单显得过重。
- 让高频操作保持近手，低频或未来能力保持可发现。
- 保持 macOS 桌面应用习惯，同时兼容 MyNote 当前应用内交互模式。

## 3. 非目标

本轮不做以下内容：

- 逐项定义每个菜单项的最终快捷键。
- 逐项定义所有未来功能的交互细节。
- 多窗口完整方案。
- 工具栏、命令面板、顶部按钮组的最终视觉设计。
- Windows / Linux 的原生菜单差异处理细则。

这些内容应在后续实现或平台适配阶段继续细化。

## 4. 设计结论

本轮确认采用以下骨架：

- 系统菜单主结构：文件、编辑、视图、笔记、帮助。
- 不单独设立“组织”菜单。
- 不把“窗口”作为本轮主设计重点；macOS 原生窗口项可保留为系统默认薄菜单，但不承载业务能力。
- 右键菜单按对象分型，而不是做一个全局通用大菜单。
- 未来功能可灰态预留，但必须克制，只保留真正稳定的结构位。

这是一个“平衡型双轨方案”——系统菜单提供全局一致入口，右键菜单提供近手对象操作，二者互相补充而不互相替代。

## 5. 菜单设计原则

### 5.1 系统菜单负责全局一致动作

系统菜单只承载以下类型的动作：

- 跨页面、跨对象都成立的全局能力。
- 用户天然会去顶部菜单寻找的标准桌面动作。
- 具备明确快捷键语义的动作。
- 会影响整体工作模式的视图切换。

不应把强依赖当前对象状态的动作硬塞进系统菜单一级结构。

### 5.2 右键菜单负责对象化动作

右键菜单只承载以下类型的动作：

- 强依赖当前节点、标签、选区或空白区上下文的动作。
- 需要就地操作、就地反馈的动作。
- 不适合作为全局菜单主项展示的整理操作。

不同对象必须拥有不同菜单，不允许“任何地方都弹出同一长菜单”。

### 5.3 先服务当前高频，再安置未来能力

高频动作必须直出：

- 新建笔记。
- 重命名。
- 移动。
- 查找替换。
- 侧栏显隐。
- 标签筛选 / 上下文查看。

未来能力只在其信息架构位置稳定时才灰态预留，例如：

- 图谱视图。
- 修订记录。
- 知识关系。
- 知识库导出。

### 5.4 命名必须面向用户动作

菜单命名优先采用用户动作语言，而不是内部实现语言，例如：

- 使用“移动到笔记本”而不是“更新路径”。
- 使用“显示反向链接”而不是“打开链接关系查询”。
- 使用“管理标签”而不是“标签治理”。

## 6. 系统菜单结构

### 6.1 文件

文件菜单用于承载知识库级与文档级输入输出动作。

建议菜单项：

1. 新建知识库
2. 打开知识库
3. 打开最近知识库
4. 关闭知识库
5. 新建笔记
6. 导入 Markdown
7. 导出当前笔记
8. 导出知识库（灰态预留）
9. 设置

设计原因：

- “知识库生命周期”与“文件输入输出”都属于文件语义。
- 新建笔记虽然也属于业务能力，但用户在桌面应用中常会从文件菜单寻找“新建”。
- 设置放在文件菜单末尾，对当前产品阶段是可接受的简化；后续若设置复杂度上升，可再迁移到应用菜单或独立偏好设置入口。

不建议放入文件菜单的内容：

- 侧栏显隐。
- 标签、关系、图谱等结构性功能。
- 当前对象专属操作，如复制 Wiki 链接。

### 6.2 编辑

编辑菜单用于承载标准文本编辑动作与与选区强相关的通用插入能力。

建议菜单项：

1. 撤销
2. 重做
3. 剪切
4. 复制
5. 粘贴
6. 全选
7. 查找
8. 替换
9. 插入链接
10. 插入标签
11. 粘贴图片为附件

设计原因：

- 这组动作天然和编辑器行为相关。
- 文本编辑标准动作应与系统预期一致。
- “插入链接 / 插入标签”虽然偏业务，但本质仍是对文本内容的编辑增强。

不建议放入编辑菜单的内容：

- 显示反向链接。
- 打开图谱。
- 移动笔记。
- 管理标签或管理关系。

### 6.3 视图

视图菜单用于承载界面显隐和工作模式切换，是当前骨架中必须保留的一组。

建议菜单项：

1. 显示/隐藏左侧栏
2. 显示/隐藏右侧栏
3. 编辑模式
4. 分屏预览
5. 专注模式
6. 搜索面板
7. 图谱视图（灰态预留）
8. 修订记录面板（灰态预留）

设计原因：

- 这些动作改变的是用户的工作视角，而不是内容本身。
- 侧栏显隐、编辑模式切换、专注模式都具备强烈的“视图”语义。
- 图谱和修订如果未来以“面板/视图”方式进入主界面，就应预留在视图菜单而不是文件或笔记菜单。

不建议放入视图菜单的内容：

- 重命名笔记。
- 删除笔记。
- 管理标签。
- 导入导出。

### 6.4 笔记

笔记菜单用于承载当前打开笔记或当前选中笔记的业务动作，是原本“组织”菜单内容的收敛容器。

建议菜单项：

1. 重命名当前笔记
2. 移动到笔记本
3. 复制笔记链接
4. 显示反向链接
5. 管理标签
6. 创建关联（灰态预留）
7. 查看知识关系（灰态预留）
8. 删除笔记
9. 恢复笔记（按场景显示）

设计原因：

- 这组动作都围绕“当前笔记对象”展开。
- 将其集中放在笔记菜单中，能避免把结构性动作拆散到文件、编辑、视图。
- 未来知识关系功能若以“笔记与笔记之间的业务关系”为核心，放在笔记菜单比新开“组织”菜单更自然。

### 6.5 帮助

帮助菜单用于承载说明、帮助文档与应用信息。

建议菜单项：

1. 欢迎页 / 快速开始
2. Markdown 语法帮助
3. 快捷键列表
4. 检查更新（未来）
5. 关于 MyNote

设计原因：

- 当前产品仍处于功能持续演进阶段，帮助入口应保持可见。
- Markdown 语法和快捷键是高价值帮助内容。

## 7. 右键菜单对象分型

本轮建议固定 5 类右键菜单对象。

### 7.1 笔记本 / 目录节点

适用对象：左侧文件树中的顶级笔记本或目录节点。

建议菜单项：

1. 新建笔记
2. 新建子目录（灰态预留）
3. 重命名笔记本
4. 更换颜色
5. 调整顺序
6. 删除笔记本

设计原因：

- 这类对象承担容器职责，应以容器管理动作优先。
- “更换颜色 / 调整顺序”是当前已存在或已确认方向的对象化动作。

### 7.2 笔记节点

适用对象：左侧文件树中的具体笔记。

建议菜单项：

1. 打开
2. 在新窗口打开（灰态预留）
3. 重命名
4. 移动到…
5. 复制路径
6. 复制 Wiki 链接
7. 显示反向链接
8. 删除

设计原因：

- 这是最典型的对象化菜单，必须高频直出。
- 复制路径和复制 Wiki 链接分别服务不同使用场景，建议同时保留。

### 7.3 标签

适用对象：左侧标签列表中的标签项，或未来在编辑区可交互的标签表现。

建议菜单项：

1. 查看标签上下文
2. 按此标签筛选
3. 重命名标签（灰态预留）
4. 合并标签（灰态预留）
5. 移除标签

设计原因：

- 当前标签既是过滤入口，也是导航入口。
- 未来的标签治理能力可以在此平滑扩展，但当前不宜一次性暴露过多。

### 7.4 编辑器文本选择

适用对象：编辑器中选中的文本或 Markdown 片段。

建议菜单项：

1. 加粗
2. 斜体
3. 转为标题
4. 转为列表
5. 转为引用
6. 转为代码块
7. 添加链接
8. 添加标签
9. 创建双链
10. 创建知识关联（灰态预留）

设计原因：

- 这类菜单必须围绕“选中的内容怎么被转化”来组织，而不是围绕应用结构。
- “创建双链”与“创建知识关联”在心智上相近，但阶段不同，后者可先灰态。

### 7.5 空白区域

适用对象：文件树空白处、编辑区空白处、工作区未选中任何对象时的空白背景。

建议菜单项：

1. 新建笔记
2. 粘贴
3. 刷新索引
4. 显示侧栏

设计原因：

- 空白区菜单应极轻，避免把任何对象操作错误地暴露出来。
- “刷新索引”属于知识库级维护动作，放在空白区是合理的辅助入口。

## 8. 灰态预留策略

未来功能允许预留，但必须遵守以下规则：

### 8.1 只预留稳定结构位

允许灰态预留的项：

- 图谱视图
- 修订记录面板
- 创建关联
- 查看知识关系
- 导出知识库
- 重命名标签
- 合并标签
- 在新窗口打开

这些能力的对象归属和菜单位置已经清晰，即使暂未实现，也不会误导整体架构。

### 8.2 禁止灰态路线图堆积

不允许在菜单里同时堆出大量尚未开始设计的未来项，例如：

- AI 总结
- AI 写作
- 批量知识清洗
- 云同步管理
- 模板中心

这类能力应在真正进入设计阶段后再决定入口位置。

### 8.3 灰态文案必须明确

灰态项展示时应采用统一规则，例如：

- 菜单项可见但不可点击。
- 文案后附“即将推出”或使用统一占位提示。
- 不出现模糊占位词，例如“高级功能”“更多工具”。

## 9. 快捷键与菜单关系

系统菜单中的高频动作应尽量具备快捷键，右键菜单则主要强调就地操作。

建议优先绑定快捷键的动作：

- 新建笔记
- 查找
- 替换
- 显示/隐藏左侧栏
- 显示/隐藏右侧栏
- 编辑模式 / 分屏预览切换
- 专注模式

建议主要留在右键菜单、不必强调快捷键的动作：

- 更换笔记本颜色
- 调整顺序
- 复制 Wiki 链接
- 查看标签上下文
- 刷新索引

原则是：

- 高频全局动作适合快捷键。
- 强对象动作优先右键，快捷键不强求。

## 10. 分阶段落地建议

### 第一阶段：与现有能力对齐

优先落地：

- 文件、编辑、视图、笔记、帮助五组系统菜单骨架。
- 笔记本、笔记、标签、空白区四类右键菜单。
- 与当前已实现能力一致的可点击项。

### 第二阶段：补齐编辑器对象菜单

补齐：

- 编辑器选区右键菜单。
- Markdown 结构转化命令。
- 插入链接、添加标签、创建双链的快捷入口。

### 第三阶段：接入未来能力

逐步激活灰态项：

- 图谱视图。
- 修订记录面板。
- 知识关系创建与查看。
- 导出知识库。
- 标签治理。
- 多窗口相关入口。

## 11. 风险与约束

### 11.1 风险：系统菜单与右键菜单重复过多

如果没有执行层面的克制，同一动作会在顶部菜单、按钮、右键菜单同时出现，造成噪音。

控制方式：

- 顶部菜单保留全局入口。
- 右键菜单保留近手入口。
- 页面按钮只保留最高频动作，不做完整复制。

### 11.2 风险：灰态项削弱完成度感知

过多灰态会让产品显得像路线图而不是可用软件。

控制方式：

- 灰态数量严格限制。
- 每组菜单最多保留 1 至 2 个灰态关键位。

### 11.3 风险：对象分型不彻底

如果后续实现时偷懒复用同一菜单，会破坏本设计最关键的心智结构。

控制方式：

- 实现时明确区分对象来源。
- 每个对象菜单单独维护配置或构建函数。

### 11.4 约束：当前不把“窗口”作为主业务菜单

macOS 上可以继续保留系统原生窗口菜单能力，但不在本设计中承载 MyNote 业务动作。若未来引入多窗口编辑，再单独扩展窗口层设计。
