## 1. 背景

MyNote 已经完成 notebook management 的后端与前端功能接线，当前顶级笔记本支持：

- 重命名
- 修改图标和颜色
- 删除空笔记本
- 上移 / 下移排序

但现有前端交互仍然偏重：

- 依赖每行右侧“三点”作为唯一入口
- 点击后出现较大的操作面板
- 操作面板把重命名、颜色编辑、排序、删除全部集中到一块
- 用户要先开菜单，再开二级面板，交互链条过长

用户已明确要求把当前这套“三点 + 弹框”整体替换掉，改成更直接的行内交互：

- 上下移动用两个三角形表示
- 改名称直接双击文字
- 改颜色直接单击标题前面的竖条
- 删除直接用圆形叉

## 2. 目标

本轮设计目标：

- 去掉顶级笔记本上的三点按钮。
- 去掉当前白色大操作面板。
- 把重命名改成双击标题直接编辑。
- 把改颜色改成单击标题前竖条后直接出现行内色块选择。
- 把排序改成两个小三角按钮表达上移和下移。
- 把删除改成圆形叉按钮表达直接删除动作。
- 保持未归档、普通子目录、普通文章不出现这些 notebook 管理操作。
- 保持后端 notebook management 能力和约束不变。

## 3. 非目标

本轮不做以下内容：

- 不修改笔记本后端数据模型。
- 不改变“只允许删除空笔记本”的业务规则。
- 不改为拖拽排序。
- 不增加批量管理。
- 不把管理能力开放给未归档、普通目录或普通文章。
- 不再支持在 notebook 管理入口中编辑图标。
- 不在本轮重做整个左侧栏壳层样式。

## 4. 方案比较

### 方案 A：纯行内微交互（推荐）

设计：

- 去掉三点和大面板。
- 双击标题直接重命名。
- 单击竖条弹出一排小色块。
- 右侧直接显示上移三角、下移三角、删除圆叉。

优点：

- 最符合用户给出的目标图。
- 操作路径最短。
- 不再需要“先开入口，再选动作”的中间步骤。

缺点：

- 单行可操作元素变多，需要控制显隐节奏。

### 方案 B：半行内操作带

设计：

- 去掉三点，但点击笔记本后在该行下方展开一条窄操作带。
- 操作带里放颜色、排序、删除。
- 重命名仍为双击标题。

优点：

- 布局更稳，不容易把标题行挤满。

缺点：

- 仍然保留“打开一个操作区”的感觉，与用户想要的直接操作不完全一致。

### 方案 C：极简 hover 工具条

设计：

- 标题双击重命名。
- 竖条点击改颜色。
- 排序和删除按钮只在 hover 当前 notebook 行时显示。

优点：

- 默认界面最干净。

缺点：

- 发现性偏弱。
- 在桌面端可接受，但不如方案 A 直接。

本轮采用方案 A。

## 5. 最终方案

本轮前端交互采用“直接行内操作”方案，具体原则如下：

- 顶级 notebook 行不再显示三点入口。
- 顶级 notebook 行不再显示当前大操作面板。
- notebook 标题前仅保留一个竖条色标，作为颜色编辑入口。
- 当前方形字母图标不再作为行内展示元素，也不再提供编辑入口。
- 标题文本双击进入重命名输入态。
- 行尾显示三个直接动作：上移三角、下移三角、删除圆形叉。
- 这些动作默认弱化显示，在 hover 当前 notebook 行或该行处于活动态时增强可见性。
- 颜色选择采用贴近当前行的微型色块面板，选中后立即保存并自动收起。

## 6. 详细交互设计

### 6.1 适用对象

只有满足以下条件的节点使用新交互：

- `is_dir = true`
- `path` 为 `notes/<top-level-dir>`
- `path` 不为 `notes/__unarchived__`

未归档、普通子目录、普通文章继续维持当前非管理态展示。

### 6.2 标题重命名

交互规则：

- 双击顶级 notebook 标题文本后，标题切换为单行输入框。
- 输入框默认值为当前 notebook 名称。
- `Enter` 保存。
- `Escape` 取消并恢复原标题。
- 失焦时执行保存。

保存行为：

- 调用现有 `renameNotebook(oldPath, newName)`。
- 成功后退出编辑态。
- 失败后保留输入框和草稿，并在该 notebook 行附近显示错误文案。

### 6.3 颜色编辑

交互规则：

- 单击 notebook 标题前的竖条色标。
- 在当前 notebook 行下方展开一条紧凑色块条。
- 色块条只展示现有预设颜色，不展示图标选择。
- 点击某个颜色后立即调用现有 `updateNotebookVisual(notebookPath, currentIcon, color)`。
- 保存成功后色块条自动收起。
- 保存失败后保留色块条并显示错误文案。

显示规则：

- 同一时间只允许一个 notebook 的颜色条展开。
- 颜色条属于行内附属元素，不使用大卡片面板样式。

### 6.4 排序操作

交互规则：

- 在 notebook 行右侧直接显示“上移三角”和“下移三角”两个小按钮。
- 上移按钮表达为向上三角。
- 下移按钮表达为向下三角。

行为规则：

- 点击上移，按当前顶级 notebook 顺序构造新的 `orderedPaths`，调用 `reorderNotebooks`。
- 点击下移，按当前顶级 notebook 顺序构造新的 `orderedPaths`，调用 `reorderNotebooks`。
- 第一个 notebook 的上移按钮禁用。
- 最后一个 notebook 的下移按钮禁用。
- 未归档不参与该组排序按钮展示。

### 6.5 删除操作

交互规则：

- 在 notebook 行右侧直接显示圆形叉按钮。
- 叉按钮使用删除语义颜色，但默认状态不要过度抢眼。

行为规则：

- 点击后不再打开旧大面板。
- 直接在当前 notebook 行下方展开一个很小的确认条，文案为“确认删除该笔记本？”。
- 用户再次确认后调用 `deleteNotebook(notebookPath)`。
- 成功后确认条关闭。
- 失败时保留确认条，并显示例如“只能删除空笔记本”这类错误原因。

说明：

- 这里保留一次轻量确认，避免删除按钮过于危险。
- 该确认条是局部确认，不是当前那种多功能大面板。

### 6.6 可见性规则

为了避免行内控件过多带来噪声，采用以下规则：

- 默认状态下，排序与删除按钮为弱显。
- 当前 notebook 行 hover 时，排序与删除按钮增强显示。
- 当前 notebook 行处于编辑态、颜色选择态或删除确认态时，操作按钮保持可见。
- 篇数 badge 保留，但需要与右侧按钮区重新排布，避免挤压标题。

## 7. 组件与状态调整

### 7.1 FileTreeNode

需要从当前“目录按钮 + 更多操作按钮 + 注入式面板”结构，调整为：

- 标题区支持双击触发重命名。
- 竖条支持点击触发颜色选择。
- 行尾支持直接渲染三角排序按钮和圆叉删除按钮。
- 去掉三点按钮。
- 去掉对旧 menu 模式的依赖。

### 7.2 FileTreePanel

需要从当前 `menu / rename / visual / delete` 多面板状态，收敛为更轻量的状态：

- 当前重命名中的 notebook path
- 当前颜色条展开的 notebook path
- 当前删除确认中的 notebook path
- 对应草稿与错误信息

不再需要旧的“更多操作面板”状态机。

### 7.3 复用现有能力

以下前端能力继续复用：

- `renameNotebook`
- `updateNotebookVisual`
- `deleteNotebook`
- `reorderNotebooks`

后端无需新增接口。

## 8. 错误处理与边界

需要显式处理：

- 双击重命名后输入为空。
- 新名称与已有 notebook 冲突。
- 删除非空 notebook 失败。
- 颜色更新失败。
- 排序失败。
- 当前 notebook 在操作过程中因为树刷新发生位置变化。

处理原则：

- 出错时保留当前局部交互态和草稿。
- 错误文案直接贴近当前 notebook 行显示。
- 不再回退到旧的大面板交互。

## 9. 测试与验证策略

前端测试至少覆盖：

- 顶级 notebook 不再显示三点按钮。
- 双击标题进入重命名态。
- `Enter`、`Escape`、失焦保存/取消行为正确。
- 单击竖条打开颜色条。
- 点击颜色会调用 `updateNotebookVisual`。
- 上移三角与下移三角正确触发 `reorderNotebooks`。
- 首项上移禁用，末项下移禁用。
- 圆叉点击后出现轻量确认条。
- 删除失败时错误文案就地显示。
- 未归档、普通目录、普通文章不出现这些管理操作。

验证命令沿用当前仓库基线：

- `corepack pnpm vitest run src/components/LeftSidebar/FileTreePanel.test.tsx`
- `corepack pnpm build`

## 10. 风险与兼容性

主要风险：

- 行内元素增多后，标题、篇数、操作按钮之间可能拥挤。
- 双击重命名与单击展开/收起目录需要避免冲突。
- 颜色条与删除确认条都挂在行下方时，需要确保同一时间只有一个局部交互打开。

兼容性策略：

- 仅替换顶级 notebook 管理入口，不改变普通文件树节点的交互。
- 继续沿用现有后端接口与业务规则，降低回归风险。
- 若空间不足，优先压缩按钮视觉占位，不回退到大面板。
