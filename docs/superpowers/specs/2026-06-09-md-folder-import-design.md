# MyNote Markdown 文件夹导入设计

> 实现状态：当前左侧栏“导入笔记”仅支持多选单个 Markdown 文件；本设计定义文件夹导入、目录层级保留、图片附件复制与错误处理规则。

## 1. 背景

当前左侧栏文件树中的“导入笔记”入口只能选择单个或多个 Markdown 文件，不能直接导入一个包含多层目录的资料文件夹。这会带来三个直接问题：

- 用户从外部知识资料库迁移到 MyNote 时，必须手动逐层进入目录挑选 `.md` 文件。
- 原始资料目录中的相对引用图片无法随导入动作一起落入知识库，导入后容易出现图片丢失。
- 现有导入链路只覆盖“单文件复制 + 建索引”，不能表达“整组资料迁移”的行为语义。

MyNote 已经明确采用“文件优先”的知识管理策略，Markdown 文件是权威载体，附件文件应保持可迁移和可离线访问。因此，文件夹导入应优先保持原始资料的目录关系和相对资源可用性，而不是把内容拆散后重新组织。

## 2. 目标

本轮设计目标如下：

- 让左侧栏导入入口同时支持 Markdown 文件和文件夹导入。
- 导入文件夹时，递归导入其内部所有 Markdown 文件。
- 导入文件夹时，保留原始目录层级，并挂接到用户选定的目标目录下。
- 对 Markdown 中引用的本地相对图片附件进行一并复制，保证导入后图片继续可用。
- 对冲突、缺失资源、不可导入项目给出明确反馈，而不是静默失败。
- 保持现有单文件导入能力可继续工作，不破坏已有菜单和调用路径。

## 3. 非目标

本轮不做以下内容：

- 不支持导入任意非 Markdown 文档类型并自动转 Markdown。
- 不做 PDF、Office 文档、音频、视频等通用附件的完整迁移策略。
- 不做导入预览树的复杂交互，例如逐项勾选排除某些子目录。
- 不做跨知识库批量合并策略。
- 不在本轮引入“统一迁移到 assets/images”这类全局附件重写方案。

## 4. 设计结论

本轮采用“前端统一发起、后端统一展开”的方案：

- 前端导入入口支持两类来源：单个或多个 Markdown 文件、单个文件夹。
- 文件夹导入由后端递归扫描和落盘，前端不承担目录遍历与路径改写逻辑。
- 导入文件夹时，默认保留所选目录本身作为一级目录，再保留其内部相对层级。
- 本地相对图片附件只要位于所选目录内部，就随 Markdown 一起复制并保持相对位置。
- 外链图片、网络图片、或指向所选目录外部的本地文件，不自动复制，只在结果中给出警告。

用户在本轮已确认的核心规则是：保留原目录层级到目标目录下。

## 5. 导入对象模型

导入来源分为两类：

### 5.1 文件导入

- 输入是一组绝对文件路径。
- 仅接受 `.md` 文件。
- 每个文件独立导入到用户选择的目标目录。
- 保持现有导入行为与兼容性。

### 5.2 文件夹导入

- 输入是一个绝对目录路径。
- 后端递归扫描该目录下所有 `.md` 文件。
- 扫描结果中的每个 Markdown 文件都带有“相对于所选目录父层级”的目标相对路径。
- 所选目录名本身保留为目标目录下的一级目录。

示例：

- 用户选择源目录 `project`
- 目标目录为 `notes/work`
- 源文件 `project/docs/a.md`
- 最终落盘为 `notes/work/project/docs/a.md`

## 6. 交互流程

左侧栏文件树中的导入入口保持单一按钮，不拆成多个入口，但交互允许用户明确选择“文件”或“文件夹”。建议流程如下：

1. 用户点击“导入笔记”。
2. 前端弹出导入来源选择，至少包含“导入 Markdown 文件”和“导入文件夹”两个动作。
3. 用户完成选择后，弹出导入确认对话框。
4. 对话框显示：
   - 导入来源类型。
   - 来源名称。
   - 预计导入的 Markdown 文件数量。
   - 目标目录选择。
   - 如果存在可提前发现的图片附件，也可显示附件数量；若成本过高，可首版在结果里反馈，不强制预览。
5. 用户确认后执行导入。
6. 导入完成后刷新文件树，并在有成功导入笔记时打开最后一篇导入成功的笔记。
7. 若存在部分失败或警告，对话框保留结果列表而不立即关闭。

这里不建议直接把文件和文件夹混在同一个系统文件选择器里完成，因为跨平台能力和返回值一致性较弱。单一按钮下的二次动作选择更稳定，也更容易测试。

## 7. 导入落盘规则

### 7.1 目标目录规则

- 若用户未显式输入目标目录，则沿用现有默认值 `notes`。
- 若用户选择现有笔记本目录，则导入内容挂到该目录下。
- 若用户输入新目录，则先创建目录，再执行导入。

### 7.2 文件夹落盘规则

- 所选文件夹名保留为目标目录下一级目录。
- 该文件夹内部所有子目录继续按相对路径落盘。
- 空目录不单独创建，只有在包含被复制的 Markdown 或附件时才创建对应目录。

### 7.3 索引规则

- 每个导入成功的 Markdown 文件都按现有索引流程建索引。
- 文件夹导入本质上是多个 Markdown 文件导入的事务性批处理，但首版允许“部分成功、部分失败”。
- 批量导入结束后返回汇总结果，而不是只返回单个笔记对象。

## 8. 图片附件处理规则

### 8.1 处理范围

仅处理 Markdown 正文中以相对路径引用、且实际文件位于所选导入目录内部的本地图片资源，例如：

- `![cover](./images/cover.png)`
- `![diagram](../assets/diagram.jpg)`

### 8.2 复制规则

- 若图片文件位于所选导入目录内部，则一并复制到知识库目标位置。
- 图片复制时保持与当前 Markdown 文件的相对位置关系不变。
- 若目录整体被保留，且图片也按原层级复制，则 Markdown 内原始相对路径通常无需改写。

示例：

- 源目录结构：`project/docs/a.md` 与 `project/docs/images/p1.png`
- 目标目录：`notes/work`
- 落盘结构：`notes/work/project/docs/a.md` 与 `notes/work/project/docs/images/p1.png`
- `a.md` 中的 `./images/p1.png` 无需改写，导入后继续有效。

### 8.3 不自动复制的情况

以下情况不自动复制：

- `http://` 或 `https://` 网络图片。
- `data:` 内联图片。
- 指向导入目录外部的本地相对路径。
- 绝对路径本地图片。
- 非图片型链接目标。

这些项目不阻断 Markdown 导入，但要在导入结果中报告为警告。

### 8.4 为什么不迁移到统一 assets 目录

本轮不采用统一迁移到知识库 `assets/images` 的方案，原因如下：

- 会强制重写 Markdown 中的图片路径。
- 同一批导入资料中的相对目录关系会被打散。
- 多篇笔记共享相对附件目录时，迁移和回溯更复杂。
- 与“文件优先、资料原样迁移”的设计原则不一致。

## 9. 冲突与错误处理

### 9.1 Markdown 文件冲突

- 若目标路径不存在，直接复制。
- 若目标路径已存在，则沿用现有冲突重命名策略，例如 `a-1.md`、`a-2.md`。

### 9.2 图片附件冲突

- 若目标图片路径不存在，直接复制。
- 若目标图片路径已存在且内容相同，可直接复用，不重复复制。
- 若目标图片路径已存在但内容不同，则采用重命名策略生成新文件名。
- 如果某篇 Markdown 中图片文件因冲突被重命名，必须只改写该篇受影响 Markdown 内对应的目标路径。

### 9.3 部分失败策略

- 单个文件导入失败，不应导致整批导入全部回滚。
- 最终返回成功项、失败项、警告项三类结果。
- 前端结果面板至少要能展示：源路径、失败原因或警告原因。

### 9.4 常见错误类型

- 所选目录中没有任何 Markdown 文件。
- Markdown 文件无法读取。
- 图片文件引用存在但源文件缺失。
- 路径越界或非法路径。
- 知识库未打开。
- 数据库不可用。

## 10. 前后端职责划分

### 10.1 前端职责

- 触发“导入文件”或“导入文件夹”选择流程。
- 展示导入确认对话框。
- 收集目标目录输入。
- 调用新的批量导入接口。
- 渲染结果摘要和错误列表。
- 在存在成功项时刷新树并打开最后一个成功导入的笔记。

### 10.2 后端职责

- 校验输入路径与知识库状态。
- 对文件夹执行递归扫描。
- 识别 Markdown 文件与需要复制的本地图片附件。
- 负责目标路径计算、冲突处理、必要时的 Markdown 引用改写。
- 执行文件复制与索引写入。
- 返回结构化批量结果，而不是单一笔记对象。

### 10.3 接口演进建议

- 保留现有单文件 `import_note` 接口以兼容当前调用。
- 新增面向文件夹与批量结果的导入接口，例如 `import_markdown_sources`。
- 新接口输入应显式区分来源类型，避免把文件和目录通过模糊字段混传。

## 11. 测试策略

### 11.1 前端测试

- 验证导入入口能触发“文件导入”与“文件夹导入”两个分支。
- 验证确认对话框能正确展示来源类型和目标目录。
- 验证部分失败时错误列表保留，成功时触发树刷新和打开最后导入笔记。

### 11.2 后端测试

- 导入单个 Markdown 文件仍保持原行为。
- 导入文件夹时递归发现所有 Markdown 文件。
- 导入文件夹时保留目录名与相对层级。
- Markdown 中相对图片位于所选目录内部时会一并复制。
- 指向目录外部或网络图片时只报警告，不阻断导入。
- 文件与图片冲突时按规则重命名并在必要时改写引用。
- 空目录或无 Markdown 目录返回可理解的结果。

### 11.3 验证优先级

优先验证后端导入规则，因为目录保留、图片复制、冲突重命名都以此为准；前端主要验证状态流与结果展示。

## 12. 风险与约束

- 不同平台文件选择器对“同时选文件和目录”的支持不一致，因此首版不依赖混合选择器。
- Markdown 中图片目标的写法可能包含空格、编码字符或引用格式变体，路径提取必须复用现有 Markdown 解析能力，不能靠脆弱字符串替换。
- 大目录导入可能耗时较长，后续可能需要进度反馈；本轮可先完成结果式反馈，不强制引入实时进度。
- 目录外部资源不自动复制意味着部分导入后仍可能出现“图片不可用”的警告，这是有意保守策略，不应静默吞掉。
- 本规格只覆盖图片型本地附件；若后续要扩展到 PDF、音频等资源，应基于同一导入骨架继续演进。
