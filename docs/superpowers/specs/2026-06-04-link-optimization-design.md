## 1. 背景

当前工程已经具备三类相邻但割裂的能力：

- Markdown / Wiki link 自动解析与目标解析。
- 传出链接与反向链接展示。
- 手动关系的创建、查看与删除。

现状问题不在于底层能力缺失，而在于产品心智与交互入口没有统一，具体表现为：

1. 搜索结果中涉及链接的点击行为不稳定，用户无法形成可靠的跳转预期。
2. 双向链接与手动关系并存，但界面没有解释它们分别解决什么问题。
3. 编辑区插入链接仍然偏命令式，写作过程中的操作成本偏高。
4. 知识关联的价值没有在 UI 中被明确表达，导致功能存在但意义不清。

## 2. 目标

1. 明确双链与手动关系的长期共存模型，避免后续重复建设。
2. 统一链接相关点击、跳转、定位与打开笔记的行为规则。
3. 降低编辑区插入链接的操作成本，补齐右键入口。
4. 把当前分散在多个面板和菜单中的关联能力组织成一致的用户心智。
5. 为后续实现提供明确的分阶段落地顺序，优先修复可用性，再整理信息架构。

## 3. 非目标

本轮只定产品和交互边界，不直接扩展以下范围：

- 全局知识图谱可视化。
- 自定义关系类型。
- 基于 AI 的自动建链建议。
- 跨知识库链接。
- 链接批量修复工作台。

## 4. 核心设计结论

### 4.1 采用双轨并行模型

知识关联长期采用“双轨并行”模型，而不是二选一：

- 双链负责轻量引用、上下文串联和快速导航。
- 手动关系负责表达明确语义，如前置、扩展、支持、对立。

二者长期共存，不做互相替代。

### 4.2 对用户只呈现一套统一心智

虽然底层上是 links 和 relations 两种数据源，但对用户的一级概念统一为“知识关联”。

在知识关联之下，再区分两类：

- 自动关联：由正文中的 Markdown link / Wiki link 自动产生。
- 手动关系：由用户显式补充的结构化关联。

### 4.3 搜索中的链接命中默认打开源笔记并定位

针对用户已确认的行为，搜索结果中如果命中的是一个链接，不默认直接打开目标笔记，而是：

1. 打开当前命中的源笔记。
2. 定位并高亮该链接所在位置。
3. 用户再决定是否点击该链接跳向目标。

该规则比“直接跳目标”更符合搜索语义，因为搜索命中的是“在哪里提到了这个链接”，而不是“直接去目标”。

## 5. 用户价值定义

链接能力的价值不是“支持链接语法”本身，而是为用户提供以下三类收益：

1. 写作价值：在记录过程中低成本把一条知识与另一条知识挂接起来，减少孤立笔记。
2. 阅读价值：在查看一条笔记时，快速知道它引用了谁、谁引用了它、它与哪些知识存在明确关系。
3. 复盘价值：在持续积累后，用户可以围绕某个主题看到上下文、来源、延伸与支撑，逐步形成结构化知识网络。

因此，后续所有实现都应围绕“降低挂接成本”和“提升回看收益”两条主线展开。

## 6. 信息模型

### 6.1 自动关联

自动关联包括：

- 传出链接：当前笔记正文提到的目标笔记或外部地址。
- 反向链接：其他笔记正文中提到当前笔记的记录。

自动关联由现有 link extraction 和 reconciliation 链路维护，不要求用户手动双写。

### 6.2 手动关系

手动关系继续使用现有关系模型，如：

- 相关
- 前置
- 扩展
- 支持
- 对立
- 相似

手动关系的价值是补足“为什么相关”，而不是替代“提到了谁”。

### 6.3 双向链接的产品解释

双向链接在产品上的解释固定为：

- 用户只需要在 A 中链接 B。
- 系统自动让 B 获得一条来自 A 的反向链接。
- 用户不需要手动维护两端。

也就是说，双向链接是“单次写入、双向可见”的导航能力，而不是一套独立于链接之外的新关系系统。

## 7. 交互设计

### 7.1 一级入口统一为“关联”

右侧边栏不再让 links 与 relations 呈现为两套割裂能力，而是整合为统一的“关联”工作面。

推荐的分区顺序：

1. 提及了谁：传出链接。
2. 谁提及我：反向链接。
3. 关系：手动关系。

这样可以先满足阅读与导航，再补充结构化语义。

### 7.2 不改变底层数据权威性

整合只发生在 UI 心智层，不改变当前底层权威来源：

- links 仍来自 Markdown / Wiki link 解析。
- relations 仍来自 SQLite 手动关系。

### 7.3 保留上下文菜单，但不把菜单当主入口

上下文菜单继续承担快捷操作职责，但不再承担主要发现路径。用户首先应在右侧“关联”面板理解系统提供了哪些关联能力，再通过菜单加速操作。

## 8. 搜索与跳转规则

### 8.1 点击规则统一

所有内部笔记类跳转都应最终走统一的打开笔记路径，避免不同来源行为不一致。来源包括：

- 搜索结果
- Markdown 预览中的内部链接
- 右侧关联面板中的传出链接
- 右侧关联面板中的反向链接
- 手动关系项

### 8.2 搜索命中为链接时的行为

当搜索结果命中的是“某条链接出现的位置”，点击后执行：

1. 打开源笔记。
2. 滚动到对应 source line / 匹配区间。
3. 在编辑区或预览区给出可见高亮。

这个动作的语义是“带我去看命中的上下文”。

### 8.3 真正跳往目标的行为

当用户在源笔记的预览、编辑器辅助交互或关联面板中再次点击具体链接时，才执行“打开目标笔记”。

这个动作的语义是“沿链接继续导航”。

### 8.4 外部链接行为

外部链接继续保持直接打开外部地址，不引入额外中转。

## 9. 编辑区插入方案

### 9.1 新增右键入口

编辑区右键菜单应成为低成本插入入口，至少包含以下动作：

- 转为双链
- 转为 Markdown 链接
- 插入链接到笔记...
- 建立关系...

### 9.2 选区态与空白态分流

有选中文本时：

- “转为双链”优先出现，用选中文本作为 title 查询或初始候选。
- “转为 Markdown 链接”作为次选。

无选中文本时：

- 提供“插入链接到笔记...”并弹出目标笔记选择。
- 不强迫用户先手写语法。

### 9.3 关系创建不侵入主写作流

“建立关系...”保留在右键和右侧关联面板中，但不应成为编辑区默认最高频入口。其定位是“补语义”，不是“代替写链接”。

## 10. 右侧关联面板重构

### 10.1 结构

右侧面板统一为三块：

- 传出链接
- 反向链接
- 手动关系

### 10.2 每块支持的动作

传出链接：

- 点击打开目标。
- 右键打开上下文菜单。

反向链接：

- 点击打开来源笔记。
- 后续可扩展“定位到链接出现位置”，但首阶段不是硬要求。

手动关系：

- 点击打开目标笔记。
- 支持删除关系。
- 支持从面板内直接新增关系。

### 10.3 命名建议

面板标题建议统一使用“关联”，而不是“链接”。

原因：

- “链接”只覆盖自动关联，不覆盖手动关系。
- “关联”更贴近最终用户价值，而不是底层技术机制。

## 11. 分阶段落地计划

### 阶段一：修复可用性

目标：先消除“能看到但不能可靠使用”的问题。

实施内容：

1. 修复搜索结果中的链接点击无反应问题。
2. 统一内部笔记跳转入口，收敛到同一 open-note 路径。
3. 确保搜索命中链接时能打开源笔记并定位高亮。

### 阶段二：统一关联面板

目标：把用户心智从分裂的 links / relations 收敛到统一的关联面板。

实施内容：

1. 重构右侧栏结构和命名。
2. 统一传出链接、反向链接、手动关系的分区顺序。
3. 保留现有数据源，不做 schema 改造。

### 阶段三：降低编辑成本

目标：让用户在写作时更自然地插入链接和补关系。

实施内容：

1. 为编辑区右键菜单补齐链接插入动作。
2. 为选区提供快速“转为双链 / Markdown 链接”。
3. 为空白区域提供“插入链接到笔记...”入口。

## 12. 验收标准

1. 搜索命中链接后，点击可稳定打开源笔记并定位到该链接上下文。
2. 从预览、关联面板、关系项点击内部笔记时，行为一致且可用。
3. 用户无需手写完整语法，也能通过编辑区右键完成插入链接。
4. 右侧面板中，用户可以清楚区分“提到了谁”“谁提及我”“有什么语义关系”。
5. 文案与信息架构能够解释双向链接的作用，而不是只暴露术语。

## 13. 风险与处理

### 13.1 风险：把搜索语义做成导航语义

如果搜索结果一上来就直接打开目标笔记，用户会丢失“为什么命中”的上下文。

处理：

搜索命中链接时固定先打开源笔记并定位。

### 13.2 风险：links 和 relations 再次分裂

如果 UI 继续把 links 和 relations 作为平行系统展示，用户会把它们理解成重复功能。

处理：

一级概念统一为“关联”，二级再区分自动关联与手动关系。

### 13.3 风险：关系入口过重，打断写作

如果把“建立关系”做成和“插入链接”同等重量，会抬高普通写作成本。

处理：

默认优先支持轻量双链插入，关系创建保留为补语义动作。
