## 1. 背景与问题定义

当前 MyNote 采用单窗口工作模式：编辑器与预览区位于同一窗口（分栏或编辑模式切换）。在“讲解/授课/汇报”场景中，用户通常希望：

- 主屏保留完整编辑工作台（目录、编辑、搜索、关系、菜单等）。
- 外接投影屏仅展示干净预览，避免暴露编辑操作与侧边信息。

现有单窗口分栏模式无法满足“主屏编辑 + 副屏独立投影”的展示诉求，因此需要引入“独立预览投影窗口”能力。

## 2. 目标场景与关键约束

### 2.1 目标场景

- 场景 A：讲者在笔记编辑中实时修订，投影端同步显示最终阅读视图。
- 场景 B：讲者切换笔记时，投影端自动跳转到同一笔记。
- 场景 C：讲者在主屏滚动编辑，投影端可选择自动跟随或独立滚动。

### 2.2 关键约束

- 桌面端基于 Tauri，多窗口必须遵循现有安全与能力配置。
- 方案必须以跨平台为前提，macOS、Windows、Linux 都要支持；不能把关键能力建立在某个单一平台专属 API 上。
- 预览渲染沿用当前 `MarkdownPreview` 体系，避免分叉两套渲染逻辑。
- 对主窗口编辑体验零侵入，不能引入明显输入延迟。
- 断开副屏、窗口关闭、窗口最小化时不影响主窗口继续工作。

### 2.3 跨平台兼容基线

- `开启投影预览`、`关闭投影预览`、内容同步、只读预览、手动拖拽到第二屏，这些属于三端必须一致支持的核心能力。
- `自动移动到副屏`、`自动全屏`、`聚焦外接显示器` 这类能力属于增强能力，只能在 Tauri 当前平台 API 明确可用且行为稳定时启用。
- 任一平台若无法稳定支持增强能力，必须降级为“窗口已打开，请手动拖动到目标屏幕”，但不能影响核心投影能力。
- 任何平台判断逻辑都必须封装在统一窗口适配层中，前端业务组件不直接写 `if (macOS)` 之类的分叉控制流。

## 3. 设计目标

- 支持一键开启“投影预览窗口”（下称 Projection Window）。
- Projection Window 仅展示预览，不包含编辑控件与侧栏。
- 投影窗口自动同步：当前笔记、正文内容、导航高亮状态。
- 支持“跟随主窗口滚动”开关。
- 支持快速移动到副屏并全屏显示（首次可手动二次确认）。
- 主窗口关闭投影模式时，投影窗口可控地退出。

## 4. 非目标

- 不在本期实现“双向编辑”（投影窗口反向编辑主文档）。
- 不在本期实现“投影窗口独立主题系统/模板系统”。
- 不在本期实现“远程网络投屏协议（如 WebRTC）”。
- 不在本期实现“一主多投影窗口同时输出”。

## 5. 头脑风暴与方案对比

### 5.1 方案 A：单窗口分栏增强（伪双屏）

做法：仍是一个窗口，只把预览区拖到最大并建议系统级窗口分屏投影。

优点：
- 开发成本最低。

缺点：
- 不能真正将编辑区与预览隔离到不同屏幕。
- 讲解时仍可能暴露编辑操作与侧栏信息。

结论：不满足核心场景，淘汰。

### 5.2 方案 B：独立 Tauri 第二窗口（推荐）

做法：主窗口创建一个独立 webview 窗口，运行“投影预览壳层”，通过事件总线同步文档状态。

优点：
- 与用户心智一致，天然支持双屏。
- 主窗口与投影窗口生命周期可控。
- 兼容现有预览组件，改造边界清晰。

缺点：
- 需要补齐多窗口能力配置、窗口状态管理、跨窗口同步协议。

结论：满足目标且风险可控，推荐。

### 5.3 方案 C：系统级镜像 + 主窗口内隐藏编辑区

做法：仍单窗口，但引入“演示模式”隐藏编辑 UI，仅显示预览再配合系统镜像。

优点：
- 无需多窗口能力。

缺点：
- 主屏会丢失编辑主体，不符合“主屏继续编辑”核心诉求。

结论：仅可作为降级备选，不作为主方案。

## 6. 设计结论

采用方案 B：新增“独立投影预览窗口”。

关键设计原则：

- 主窗口是唯一编辑真源。
- 投影窗口是只读投影端。
- 同步采用“主推送 + 投影幂等应用”的事件模型。
- 投影窗口可随时重建，不影响主窗口状态。
- 先保证三端一致可用，再逐步增加平台增强能力；增强能力失败时必须自动降级。

## 7. 用户体验与交互设计

### 7.1 入口设计

新增两个入口：

1. 顶部菜单“视图”下新增：
- 开启投影预览
- 关闭投影预览
- 投影窗口跟随滚动（勾选项）

2. 编辑区标题栏新增一个轻量按钮：
- 未开启时：`开启投影`
- 已开启时：`关闭投影`

### 7.2 启动流程

1. 用户点击“开启投影预览”。
2. 系统创建 Projection Window。
3. Projection Window 加载成功后，主窗口立即推送当前快照（当前笔记路径、标题、内容、导航状态）。
4. 若检测到副屏，提示“移动到副屏并全屏”快捷操作（一次点击执行）。

这里的“移动到副屏并全屏”是增强操作，不作为跨平台核心流程的前置条件。若当前平台不支持自动定位到外接屏，则保留打开成功提示，并引导用户手动拖拽窗口到目标屏幕。

### 7.3 投影窗口内容

投影窗口只包含：

- 标题栏（可选极简）：当前笔记标题。
- 预览主体（MarkdownPreview）。
- 底部可选状态条（只读，默认隐藏）。

明确不包含：

- 左右侧栏。
- 编辑器。
- 右键编辑相关菜单。
- 可修改数据的交互控件。

### 7.4 关闭与恢复

- 关闭投影窗口：主窗口状态自动回写为未开启。
- 主窗口关闭时：投影窗口一并销毁。
- 重新开启时：自动恢复到最近一次投影设置（跟随滚动开关、是否自动全屏）。

## 8. 系统架构改造

### 8.1 窗口角色模型

新增窗口角色：

- `main`：现有主窗口，负责编辑与主状态。
- `projection-preview`：新增投影窗口，只读展示。

前端 App 启动时根据窗口标签加载不同壳层：

- `main` -> `AppShell`
- `projection-preview` -> `ProjectionPreviewShell`

### 8.2 状态分层

- 编辑真源：`useEditorStore`（主窗口）。
- 投影会话状态：新增 `useProjectionStore`（主窗口与投影窗口各自维护本地副本）。
- 跨窗口同步：Tauri event（主窗口发，投影窗口收）。

### 8.3 预览复用策略

`ProjectionPreviewShell` 复用 `MarkdownPreview`，但提供只读上下文：

- 禁用会触发编辑/跳转副作用的行为。
- 允许外链打开。
- 保留搜索/标签高亮能力（若有传入目标）。

### 8.4 同步触发点

主窗口在以下时机推送增量：

- 当前笔记切换。
- 正文内容变更（节流推送）。
- 导航目标变更（搜索定位、标签定位）。
- 跟随滚动开启时，顶部可见行变更（节流）。

## 10. Tauri 与后端改造清单

### 10.1 多窗口能力

当前 capability 仅允许 `main`，需扩展到投影窗口标签：

- 在 capability 配置中加入 `projection-preview`。
- 评估投影窗口是否需要完整权限；原则上只给最小权限集合。

### 10.2 窗口管理命令

新增命令（建议在 `commands/window.rs`）：

- `open_projection_window()`
- `close_projection_window()`
- `focus_projection_window()`
- `move_projection_to_external_monitor()`
- `set_projection_fullscreen(fullscreen: bool)`

说明：
- 可先用前端 window API 实现；若平台差异较大，再下沉 Rust 统一封装。
- 这些命令需要分成两类能力：
  - 核心能力：打开、关闭、聚焦窗口，三端必须可用。
  - 增强能力：移动到外接显示器、自动全屏，只在平台 API 可用时启用。
- 窗口 API 封装层必须返回能力探测结果，例如 `supportsExternalMonitorPlacement`、`supportsFullscreenProjection`，由 UI 决定是否展示增强操作。

### 10.4 平台适配策略

- macOS：优先支持第二窗口、手动拖拽、可选自动全屏；外接显示器定位若稳定再开启增强能力。
- Windows：优先支持第二窗口、手动拖拽、最大化/全屏；避免依赖特定 DPI 或显示器枚举假设。
- Linux：优先支持第二窗口和手动拖拽；对不同窗口管理器下不稳定的自动定位能力，默认关闭增强选项。
- 所有平台共享同一业务协议，平台差异只能存在于 `windowApi` 或 Rust 适配层，不能泄漏到 `AppShell`、`EditorWorkspace`、`ProjectionPreviewShell` 这些业务组件。

### 10.3 生命周期事件

后端向主窗口透出：

- `projection:ready`
- `projection:closed`
- `projection:error`

用于主窗口更新菜单可用状态与提示文案。

## 11. 数据与同步协议设计

### 11.1 事件通道

建议统一使用命名空间事件：

- `projection:state-sync`
- `projection:scroll-sync`
- `projection:command`

### 11.2 主状态同步载荷

`projection:state-sync` payload：

- `notePath: string | null`
- `noteTitle: string | null`
- `content: string`
- `searchNavigationTarget: SearchNavigationTarget | null`
- `tagNavigationTarget: TagNavigationTarget | null`
- `revision: number`

要求：
- 投影端按 `revision` 幂等应用，忽略旧包。

### 11.3 滚动同步载荷

`projection:scroll-sync` payload：

- `source: "main-editor" | "main-preview"`
- `topVisibleLine: number`
- `revision: number`

仅在 `projectionFollowScroll = true` 时发送。

### 11.4 同步节流

- 内容同步：200ms 节流。
- 滚动同步：80ms 节流。

## 12. 异常与降级策略

- 无外接屏：仍允许开启投影窗口，默认在主屏打开并提示手动拖拽。
- 投影窗口创建失败：主窗口保持可用，显示状态栏错误提示。
- 投影窗口崩溃/关闭：自动将 `projectionEnabled` 置 false。
- 同步超时：投影端显示“连接中断，请重试开启投影”。
- 平台不支持自动移动副屏：隐藏该快捷动作，只保留“已打开投影窗口，请拖动到目标屏幕”的提示。
- 平台支持多窗口但不稳定支持无边框全屏：回退为普通窗口或最大化窗口，不阻断投影。

## 13. 测试策略

### 13.1 前端单元测试

- 窗口角色分流：`App` 在不同角色渲染不同壳层。
- 菜单状态：开启后禁用“开启”，启用“关闭”。
- 同步协议：高 revision 覆盖低 revision。
- 跟随滚动开关：开启时发送滚动事件，关闭时不发送。

### 13.2 集成测试（Mock 窗口 API）

- 开启投影 -> 收到 ready -> 推送初始快照。
- 主窗口切换笔记 -> 投影窗口内容更新。
- 关闭投影窗口 -> 主窗口状态回收。

### 13.3 手工验收（双屏）

- macOS 外接显示器下，开启后可一键移动并全屏。
- Windows 外接显示器下，开启后至少能稳定打开独立投影窗口，并可手动拖到副屏继续同步。
- Linux 外接显示器下，开启后至少能稳定打开独立投影窗口，并可手动拖到副屏继续同步。
- 主屏编辑输入时，投影端在可接受延迟内更新。
- 断开外接显示器后应用不崩溃。

### 13.4 跨平台验证矩阵

- macOS：验证核心能力 + 增强能力。
- Windows：验证核心能力，增强能力按 API 可用性单独验收。
- Linux：验证核心能力，增强能力默认可选，不作为阻塞项。

## 14. 分阶段实施计划

### Phase 1（MVP，建议 1 个迭代）

- 打通第二窗口创建/关闭。
- 投影窗口只读渲染当前内容。
- 主窗口内容实时同步（节流）。
- 菜单基础入口可用。

### Phase 2

- 跟随滚动同步。
- 自动移动副屏与全屏辅助。
- 连接状态与异常提示完善。

### Phase 3

- 投影演讲模式增强（字号、留白、页宽、暗场模式可选）。
- 可选“隐藏元信息”与“仅正文投影”。

## 15. 验收标准

- 开启投影后，主屏保留完整编辑区，副屏仅显示预览。
- 主屏切换笔记后，投影端在 500ms 内完成切换。
- 主屏输入编辑时，投影端持续同步，无明显卡顿。
- 投影窗口关闭或异常时，主窗口不受影响。
- macOS、Windows、Linux 三端都必须支持“独立投影窗口 + 内容同步 + 手动放置到第二屏”的核心流程。
- 任一平台即使不支持“自动移动到副屏/自动全屏”，也不得影响核心投影能力上线。
- 相关单元测试与集成测试通过。

## 16. 风险与开放问题

### 16.1 风险

- Tauri 多窗口在不同平台（macOS/Windows）行为细节差异。
- 预览内容较大时，跨窗口同步可能造成渲染抖动。
- 外接显示器热插拔时窗口定位策略需要充分测试。
- Linux 窗口管理器差异较大，自动定位与全屏行为可能不一致，必须提前设计降级路径。

### 16.2 开放问题（待产品确认）

- 投影窗口是否默认显示笔记标题栏？
- 跟随滚动默认开还是关？
- 开启投影时是否自动隐藏主窗口内嵌预览，以减少重复渲染成本？

建议默认值：
- 显示标题栏：关。
- 跟随滚动：开。
- 主窗口内嵌预览：保持当前用户设置不强制更改。
