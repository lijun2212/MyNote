## 1. 背景

当前搜索功能已经具备基础 FTS 能力，但产品形态仍停留在“按笔记返回一条粗粒度结果”的阶段：

- 搜索结果仅返回笔记标题、路径与一段片段。
- 排序逻辑未被明确定义，用户无法理解为何某条结果排在前面。
- 同一篇文章内即使有多个命中位置，也只能看到一条笔记级结果。
- 点击结果后只能打开文章，不能直接跳到命中的具体位置。
- 编辑区与预览区虽然已有 source line 同步基础设施，但搜索功能尚未接入这条导航链路。

这导致搜索在“找文章”层面可用，但在“找文章里的哪一段内容”层面不可用。对知识库场景来说，这个差距会直接削弱搜索入口的价值。

本次改造目标不是重做搜索弹层，而是在现有 FTS 基础上，把搜索升级为“按命中位置导航”的入口，让搜索结果可排序、可解释、可直接落到目标段落。

## 2. 目标

本轮设计目标如下：

- 搜索必须支持全文搜索，而不是只停留在标题或摘要匹配。
- 搜索结果默认按相关度排序。
- 搜索结果按命中位置逐条展示，而不是每篇文章只保留一条结果。
- 同一篇文章内多个命中位置可以同时出现在结果列表中。
- 点击任意搜索结果后，打开对应文章并跳到该命中位置。
- 编辑区和预览区必须同步定位到同一处命中位置。
- 命中位置需要有瞬时高亮反馈，帮助用户确认跳转是否成功。
- 保留现有搜索弹层的键盘导航能力，并让它适配新的命中级结果结构。

## 3. 非目标

本轮不做以下内容：

- 高级搜索语法界面，如布尔表达式、字段过滤、日期过滤。
- 搜索结果分面导航，如按目录、标签、时间聚合。
- 搜索历史、最近搜索词、收藏搜索。
- 搜索结果分页或无限滚动优化。
- 命中词替换、批量编辑。
- OCR 专项搜索或跨知识库联合搜索。

这些能力未来都可以继续叠加，但本轮重点是“全文命中定位”与“相关度排序”。

## 4. 方案对比与结论

### 方案 A：FTS 召回 + 命中重建，推荐

先使用现有 SQLite FTS5 做全文召回和粗排序，再对召回出来的少量候选笔记读取原始 Markdown 内容，重建每一个命中位置，生成按命中位置展开的结果。

优点：

- 复用现有 `note_fts` 能力，改动集中。
- 容易拿到行号、命中片段和出现顺序，适合导航跳转。
- 不需要为本轮新增复杂的长期维护型索引表。

缺点：

- 查询链路比当前多一层命中重建。
- 需要控制候选笔记数量，避免在极端情况下放大查询成本。

### 方案 B：建立独立搜索命中索引表

在索引阶段就把正文拆分成可检索的行级或段落级命中记录，查询时直接返回命中位置。

优点：

- 查询时最直接。
- 天然适合返回命中位置和多条结果。

缺点：

- 会显著提高索引维护复杂度。
- 需要新增数据库结构和迁移，且后续更新成本更高。

### 方案 C：前端打开后再本地查找第一个命中

搜索仍然返回笔记级结果，点击后前端在文章内容里临时定位第一个匹配。

优点：

- 后端改动最少。

缺点：

- 不能支持“每个命中位置单独展示一条结果”。
- 排序无法建立在命中级相关度上。
- 同一篇文章多个命中点不可直达。

### 结论

本轮采用方案 A。

原因：

- 能满足用户明确要求的全文搜索、相关度排序、多命中逐条展示与点击后精准跳转。
- 风险和复杂度明显低于重建长期索引。
- 能最大程度复用现有编辑区/预览区联动基础设施。

## 5. 产品定义

本轮将搜索定义为：

- 知识库级全文召回入口。
- 命中位置级导航入口。
- 帮助用户快速回到文章具体段落的快捷定位工具。

搜索弹层中的单条结果不再表示“一篇文章”，而表示“某篇文章中的一个命中位置”。

因此结果列表的核心语义从：

- 标题 + 模糊片段

升级为：

- 文章标题
- 命中片段
- 命中行信息
- 文章路径
- 命中位置在整组结果中的相关度顺序

## 6. 交互与信息结构

### 6.1 查询触发

保留现有搜索弹层和输入框行为：

- 输入非空查询后，300ms 防抖触发搜索。
- 空查询时不执行搜索。
- 键盘上下键继续切换当前选中结果。
- 回车打开当前选中结果。

### 6.2 结果结构

每个结果项至少展示：

- 文章标题。
- 命中片段，片段内高亮查询词。
- 命中行号或命中位置提示。
- 文章路径。

同一篇文章允许出现多条结果；这些结果不做默认折叠，也不按文章合并。

### 6.3 点击行为

点击任意结果项后：

- 打开对应文章。
- 编辑区滚动到命中位置所在行。
- 预览区滚动到对应的同一 source line。
- 编辑区显示瞬时高亮。
- 预览区显示对应位置瞬时高亮。
- 搜索弹层关闭。

### 6.4 结果反馈

无结果时显示“未找到匹配的笔记”。

有结果时不额外展示“按相关度排序”的说明文案，但结果顺序必须稳定、可解释，不能出现明显反直觉的跳动。

## 7. 排序与相关度规则

用户已明确选择“相关度优先”，因此本轮不提供切换排序方式的 UI。

推荐采用以下排序规则，从高到低：

1. 标题命中优先于正文命中。
2. 完整词命中优先于仅前缀命中。
3. 命中片段中查询词更集中、更靠前的结果优先。
4. 同一篇文章中出现顺序更早的命中，在其他条件接近时优先。
5. 最后使用 FTS `bm25` 分数打破接近结果。

实现上不要求向用户暴露完整打分公式，但必须保证：

- 相同查询下排序稳定。
- 同一结果重复搜索时顺序基本一致。
- 标题直接命中的文章不会被无关正文长文轻易压下去。

## 8. 数据模型与接口设计

现有前端 `SearchResult` 结构不足以表达命中级导航目标，需要升级。

建议新增搜索命中导航模型：

```ts
export interface SearchNavigationTarget {
  note_id: string;
  note_path: string;
  note_title: string;
  line_start: number;
  line_end: number;
  occurrence_order: number;
  match_text: string;
  context_snippet: string;
  source: "title" | "body";
  revision: number;
}

export interface SearchResult {
  note_id: string;
  title: string;
  path: string;
  snippet: string;
  line_start: number;
  line_end: number;
  occurrence_order: number;
  match_text: string;
  source: "title" | "body";
  score: number;
}
```

其中：

- `line_start` / `line_end` 用于编辑区与预览区定位。
- `occurrence_order` 用于区分同一行内多次重复命中。
- `match_text` 用于高亮或调试。
- `source` 用于排序权重和未来展示扩展。
- `score` 用于结果排序和诊断，不要求直接展示给用户。

## 9. 前端状态与跳转机制

当前前端已有 `tagNavigationTarget`，能够驱动编辑区与预览区一起跳转并高亮。搜索导航应复用同一思路，但不应强行复用标签模型字段，以免语义混乱。

建议在 `useEditorStore` 中新增：

- `searchNavigationTarget: SearchNavigationTarget | null`
- `setSearchNavigationTarget(target)`

行为要求如下：

- 打开普通笔记时，不自动写入搜索导航目标。
- 点击搜索结果后，先通过 `openNote` 打开目标文章。
- 笔记打开完成并且请求仍然有效后，再设置 `searchNavigationTarget`。
- `MarkdownEditor` 与 `MarkdownPreview` 同时监听该目标，并按 `line_start` 滚动到指定位置。
- 高亮采用瞬时态，时间可与标签导航保持一致，默认约 1600ms。

为了避免与标签导航冲突，优先级定义如下：

- 单次点击搜索结果时，以 `searchNavigationTarget` 为当前导航来源。
- 单次点击标签上下文时，以 `tagNavigationTarget` 为当前导航来源。
- 两者不需要同时叠加生效；后触发者覆盖前触发者的瞬时定位效果。

在编辑区和预览区实现上，可以抽象出“导航目标”这一公共概念，但本轮不强制先做通用抽象，只要求行为正确。

## 10. 后端查询设计

现有 `search_notes` 使用 FTS5 返回笔记级结果。方案 A 下，推荐将查询拆成两步：

### 10.1 召回阶段

继续基于 `note_fts` 执行 FTS5 搜索，拿到候选笔记集合和粗排序分数。

候选集合建议限制在一个可控上限，例如前 20 到 50 篇，以控制后续命中重建开销。

### 10.2 命中重建阶段

对候选笔记逐篇读取 Markdown 源文，基于查询词执行命中重建：

- 找出标题命中。
- 找出正文命中。
- 计算每个命中所在行号。
- 生成带高亮标记的 `snippet`。
- 记录同一篇文章内的 `occurrence_order`。

命中重建后，将结果展开为“命中级列表”，再执行最终排序。

### 10.3 行号规则

搜索结果中的 `line_start` / `line_end` 必须基于原始 Markdown 文件行号，而不是预览去除 front matter 后的行号。

原因：

- 编辑区滚动使用原始文档行号。
- 预览区已有 front matter 行偏移翻译逻辑，可自行转换。

### 10.4 片段生成规则

片段生成要求：

- 以命中位置为中心截取局部文本。
- 命中词使用 `<mark>` 包裹，沿用现有前端安全解析逻辑。
- 不返回任意原始 HTML 标签，避免引入新的 XSS 面。

## 11. 测试策略

### 11.1 Rust 后端测试

新增或更新以下测试：

- 全文搜索可命中标题与正文。
- 同一篇文章多个命中位置会展开成多条结果。
- 标题命中排序高于普通正文命中。
- 返回结果包含 `line_start`、`occurrence_order` 与 `score`。
- 片段高亮只包裹命中词，不引入原始 HTML 污染。

### 11.2 前端 Hook 测试

更新 `useSearch` 测试：

- 支持新的结果结构。
- 旧请求不会覆盖新请求。
- KB 切换后旧结果不会污染当前结果。

### 11.3 搜索弹层测试

新增或更新以下测试：

- 多命中结果逐条渲染。
- 键盘上下选择与回车打开正确命中项。
- 点击结果后调用打开笔记并设置搜索导航目标。
- 片段中的 `<mark>` 安全解析继续有效。

### 11.4 编辑区与预览区测试

新增以下测试：

- 接收到搜索导航目标后，编辑区滚动到指定行。
- 预览区按 source line 同步滚动到对应位置。
- 搜索导航高亮在一定时间后自动消退。
- front matter 存在时，预览区仍能正确翻译行号。

## 12. 风险与后续扩展

### 12.1 风险

- 命中重建需要读取候选笔记原文，若候选集过大，可能影响搜索响应时间。
- 中文分词与 FTS 前缀搜索对“相关度”的直觉可能不完全一致，需要通过测试样例校准。
- 搜索导航与标签导航都依赖瞬时定位高亮，若状态边界处理不好，可能出现一次点击触发两套高亮残留。

### 12.2 后续扩展

后续可在本轮基础上继续演进：

- 增加按更新时间、按标题、按路径的切换排序。
- 对结果按文章分组并支持折叠展开。
- 支持高级搜索语法与筛选条件。
- 为搜索命中建立长期缓存索引，进一步降低大库查询延迟。
- 在结果列表中展示更多结构化上下文，例如所属标题层级。
