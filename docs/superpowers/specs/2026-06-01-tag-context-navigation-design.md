## 1. 背景

当前标签系统已经具备以下基础能力：

- 标签可从 Front Matter `tags` 与正文 `#标签` 中抽取。
- 左侧标签栏可列出标签并支持新增、删除、拖拽。
- 点击标签后，文件树会按标签结果切换文章集合。
- 编辑区和预览区已经具备基于 source line 的联动滚动基础设施。

现状的关键问题不是“标签没有数据能力”，而是“标签的产品表达过弱”。用户点击标签后，只能感知到文件树结果变化，却看不到这个标签具体连接了哪些文章、哪些语义位置，因此标签在体验上仍然像一个半成品筛选器，而不是知识导航入口。

本次改造目标不是重做标签系统，而是在现有标签归属能力之上，补足“标签上下文”与“按位置导航”两项能力，让标签从可增删、可筛选，升级为可进入具体知识位置的导航锚点。

## 2. 目标

本轮一阶段设计目标如下：

- 将标签定义为知识库中的主题导航入口，而不是仅用于过滤文件树的条件。
- 在左侧标签栏下部新增“标签上下文区”，用于展示选中标签相关的文章及命中位置。
- 标签上下文区默认按笔记最近更新时间倒序，仅显示最新的 5 篇命中文章。
- 超过 5 篇时显示省略提示或“查看更多”占位，不在本阶段展开完整详情页。
- 点击标签上下文区的任意文章结果后，直接打开该文章并定位到标签命中位置。
- 编辑区与预览区必须同时到达同一标签命中位置，并提供瞬时高亮反馈。
- 保留现有标签列表、新增、删除、拖拽、文件树按标签过滤等基础能力。

## 3. 非目标

本轮不做以下内容：

- 标签独立详情页。
- 标签重命名、合并、批量治理。
- 同一篇文章内多个命中点的展开选择器。
- 标签图谱、标签共现分析或标签关系网络。
- 标签访问历史、最近使用记录或复杂导航面包屑。
- 将标签上下文区扩展成完整的二级信息面板。

这些内容可作为后续阶段扩展，但不纳入本次详细设计的一阶段实现范围。

## 4. 方案对比与结论

### 方案 A：轻量侧栏上下文

在标签栏下部新增“标签上下文区”，显示选中标签的文章与命中位置。点击结果直接导航到文章中的标签位置。

优点：

- 与现有壳层和布局兼容。
- 改动聚焦，能最快建立“标签 = 导航入口”的核心体验。
- 不打断当前主工作区结构。

缺点：

- 信息密度受左栏宽度限制。
- 长期扩展空间不如独立详情页。

### 方案 B：独立标签详情页

点击标签后，主工作区切换到独立的标签详情页，左栏保留标签主列表与摘要。

优点：

- 长期扩展性最好。
- 便于承载更多标签治理和分析能力。

缺点：

- 会打断当前编辑流。
- 第一阶段实现成本明显偏高。

### 方案 C：轻量上下文 + 详情页双层模式

左栏先显示轻量上下文，点击“查看更多”后进入标签详情页。

优点：

- 兼顾当前效率和长期扩展。

缺点：

- 设计复杂度最高。
- 第一阶段容易把两个入口都做成半成品。

### 结论

本轮推荐采用方案 A。

原因：

- 与用户当前目标完全对齐。
- 可以在不重构主工作区的前提下，最直接地强化标签导航价值。
- 后续若要演进到方案 C，不会推翻本轮实现。

## 5. 产品定义

本轮将标签定义为：

- 主题导航锚点。
- 文章集合入口。
- 文章内具体语义位置的定位入口。

标签栏拆分为两层职责：

- 上半区：标签主列表与标签操作。
- 下半区：当前选中标签的上下文结果。

标签的主动作不再定义为“仅过滤文件树”，而定义为“激活当前标签上下文”。文件树切换为该标签命中的文章集合，只是这个激活动作的副作用之一，而不是唯一结果。

## 6. 交互与信息结构

### 6.1 标签栏结构

标签栏采用上下两段固定结构：

- 上半区：标签列表。
- 下半区：标签上下文区。

上半区继续负责：

- 标签显示。
- 标签高亮。
- 新增。
- 删除。
- 拖拽到编辑区。

下半区在有标签激活时显示：

- 当前标签名。
- 命中文章总数。
- 排序规则说明：按最近更新时间显示最新 5 篇。
- 5 条文章结果。
- 超过 5 条时显示省略提示。

无标签激活时，下半区显示提示态：“选择标签以查看相关文章与命中位置”。

### 6.2 结果项结构

每条结果项最少展示以下三类信息：

- 文章标题。
- 文章路径或所在目录。
- 标签命中位置摘要。

命中位置摘要优先展示：

- 所属标题上下文。
- 标签附近的简短片段。
- 若命中来自 Front Matter，则显示“Front Matter 标签”。

### 6.3 交互规则

点击标签列表中的标签后：

- 标签列表高亮当前标签。
- 文件树切到该标签命中的文章集合。
- 标签上下文区刷新为该标签的最新 5 篇文章。
- 当前打开笔记不强制切换。

点击标签上下文区的某条文章结果后：

- 打开目标文章。
- 编辑区滚动到该标签命中位置。
- 预览区滚动到对应的同一命中位置。
- 编辑区与预览区均显示瞬时高亮。

切换到另一个标签时：

- 标签上下文区整体替换为新标签结果。
- 不保留旧标签上下文内容。

删除当前激活标签时：

- 清空当前标签上下文。
- 文件树恢复非标签过滤态。

## 7. 数据模型与查询设计

现有 `tags` 与 `note_tags` 仅能回答“某篇文章是否拥有该标签”，不能回答“该标签在文章中的哪里”。

为支撑标签上下文区与按位置导航，本轮建议新增一层派生索引：`tag_occurrences`。

### 7.1 现有层

- `tags`：标签实体。
- `note_tags`：文章与标签的归属关系。

用于：

- 标签列表。
- 标签计数。
- 按标签过滤文章。

### 7.2 新增命中层

新增 `tag_occurrences` 以记录标签在文章中的具体命中点。

建议字段：

- `id`
- `note_id`
- `tag_id`
- `source`，值为 `inline` 或 `front_matter`
- `line_start`
- `line_end`
- `heading_context`
- `context_snippet`
- `occurrence_order`

职责区分如下：

- `note_tags` 负责“文章是否属于该标签”。
- `tag_occurrences` 负责“该标签在文章中的具体位置”。

### 7.3 查询规则

标签上下文查询采用以下规则：

- 先按 `tag_id` 找出所有命中文章。
- 按文章去重。
- 按文章 `updated_at DESC` 排序。
- 只取前 5 篇文章。
- 每篇文章只选择一个代表命中点。
- 优先选 `inline` 命中。
- 若无 `inline` 命中，则回退到 `front_matter`。

这样可以保证左栏结果稳定且语义上更接近正文落点。

## 8. 前端状态与跳转机制

本轮建议拆分两类状态：

### 8.1 标签上下文状态

表示“当前用户正在查看哪个标签”。

建议字段：

- `activeTagId`
- `activeTagName`
- `activeTagResultCount`
- `activeTagPreviewItems`
- `activeTagSort`

它服务于：

- 标签列表高亮。
- 标签上下文区展示。
- 文件树按标签过滤结果。

### 8.2 标签导航目标状态

表示“一次性导航到哪篇文章、哪个标签位置”。

建议字段：

- `notePath`
- `tagName`
- `lineStart`
- `lineEnd`
- `source`
- `contextSnippet`
- `revision`

其中 `revision` 用于确保多次相同目标点击时仍能触发新一轮导航。

### 8.3 跳转流程

点击标签上下文结果后的前端流程如下：

1. 打开目标笔记。
2. 写入新的标签导航目标状态。
3. 编辑区根据 `lineStart` 滚到目标行，并在行内找到目标标签 token 做瞬时高亮。
4. 预览区根据 `lineStart` 对应的 source line 滚到目标块，并对对应标签做瞬时高亮。
5. 导航完成后，恢复到原有双栏滚动同步机制。

该设计不把标签导航直接塞入现有双栏滚动同步信号，而是将其视为一个外部导航输入，再由编辑区与预览区分别消费，以避免破坏现有同步逻辑。

## 9. 后端索引与接口设计

### 9.1 保留接口

本轮继续保留：

- `list_tags`：供标签主列表使用。
- `list_notes_by_tag`：供文件树按标签过滤使用。
- 笔记打开、保存、索引更新链路。

### 9.2 新增接口

建议新增专用接口：`get_tag_context(tag_id)`。

返回结构建议如下：

```ts
type TagContextItem = {
  note_id: string;
  note_path: string;
  note_title: string;
  note_updated_at: string;
  source: "inline" | "front_matter";
  line_start: number;
  line_end: number;
  heading_context: string | null;
  context_snippet: string;
};

type TagContext = {
  tag_id: string;
  tag_name: string;
  total_notes: number;
  visible_count: number;
  has_more: boolean;
  items: TagContextItem[];
};
```

返回数据直接对应左栏下半区展示与导航，不要求前端再做全文扫描或二次计算。

### 9.3 索引更新策略

每次笔记被索引时：

- 清理该笔记旧的 `tag_occurrences`。
- 重新抽取 Front Matter 与正文中的标签命中点。
- 重建 `tag_occurrences`。

每次标签删除后：

- 相关笔记内容被改写。
- 笔记重新索引。
- 标签上下文数据与归属数据同步更新。

## 10. 实施范围与阶段切分

### 10.1 第一阶段完成标准

第一阶段完成需满足：

- 左侧标签栏新增标签上下文区。
- 点击标签后，文件树切换到标签命中的文章集合。
- 下半区显示最近更新的 5 篇文章及命中位置。
- 点击下半区任意结果后，打开文章并同时定位编辑区与预览区。
- 导航后出现瞬时高亮反馈。
- 超过 5 篇时显示省略提示或“查看更多”占位。

### 10.2 建议实施顺序

建议按以下顺序推进：

1. 后端补 `tag_occurrences` 与 `get_tag_context`。
2. 前端补标签上下文区静态展示。
3. 接通点击结果后的打开与定位。
4. 收尾高亮反馈与空态、无结果态、超量提示态。

## 11. 测试策略

### 11.1 Rust 测试

应补充：

- 标签命中抽取测试。
- `tag_occurrences` 写入与覆盖更新测试。
- `get_tag_context` 排序与去重测试。
- 每篇文章代表命中点选择规则测试。
- 删除标签后上下文数据更新测试。

### 11.2 前端测试

应补充：

- 点击标签后标签上下文区渲染测试。
- 上下文区最多展示 5 条测试。
- 超过 5 条时省略提示测试。
- 点击上下文结果后导航状态派发测试。
- 编辑区与预览区消费同一标签导航目标的测试。
- 提示态、无结果态测试。

### 11.3 端到端测试

应覆盖：

- 用户点击标签。
- 左栏显示上下文区。
- 文件树切换到命中文章集合。
- 点击上下文结果后文章打开。
- 编辑区与预览区定位到目标标签位置。

### 11.4 验收标准

本轮验收以以下五条为准：

- 点击标签后，左栏下部可看到该标签最新 5 篇命中文章。
- 文件树同步切换到该标签命中的文章集合。
- 点击任意结果，可直接导航到该标签位置。
- 编辑区与预览区可同时定位到同一命中点。
- 保存、删除标签、切换标签后，上下文与结果保持一致。

## 12. 风险与后续扩展

### 12.1 主要风险

- 现有标签抽取逻辑需要从“只返回标签名”升级为“返回标签命中集合”，索引链路会增加复杂度。
- 编辑区与预览区的标签命中高亮需要额外导航输入，不能与现有滚动同步信号混淆。
- 同一篇文章多次命中时，第一阶段必须稳定选择代表入口，否则上下文区结果会漂移。

### 12.2 后续扩展方向

后续可在本设计基础上继续扩展：

- 标签详情页。
- 标签重命名、合并。
- 同一篇文章的多命中展开。
- 标签共现分析。
- 标签图谱或标签关系视图。

这些能力不影响本轮设计成立，均可建立在当前“标签上下文 + 按位置导航”的基础之上。
