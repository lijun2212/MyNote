## 1. 背景

当前笔记本已经具备“顶层目录 = 笔记本”的产品语义，但它仍然只有名字，没有视觉身份。

这带来三个直接问题：

- 左侧笔记树的顶层目录辨识度弱，多个笔记本并列时缺少视觉分层。
- 新建笔记本流程过于工具化，只有“建一个目录”的感觉，没有“创建一个知识主题容器”的感觉。
- 后续如果要扩展笔记本排序、描述、封面或其他展示属性，当前没有稳定的元数据承载位置。

用户本轮明确提出两个要求：

- 笔记本需要允许选择有风格的图标和颜色。
- 图标和颜色应与知识库一起保存，而不是只存在当前应用实例里。

因此本次设计不是单纯给目录加一个前端样式，而是要为笔记本引入一层稳定、可持久化、可扩展的展示元数据模型。

## 2. 目标

本轮设计目标如下：

- 在“新建笔记本”流程中允许用户选择预设图标和预设颜色。
- 将笔记本图标与颜色持久化为知识库级元数据，跟随知识库一起保存。
- 侧边栏顶级笔记本节点读取并展示对应图标和颜色。
- 当元数据缺失、损坏或节点不存在时，界面必须稳定回退到默认样式，不能影响笔记树主流程。
- 为后续扩展笔记本展示属性保留统一的元数据承载位置。

## 3. 非目标

本轮不做以下内容：

- 创建后编辑笔记本图标和颜色。
- 用户上传自定义图标或输入任意色值。
- 为子目录提供独立图标和颜色配置。
- 为“未归档”分组或普通笔记文件提供图标和颜色定制。
- 笔记本封面、描述、排序、统计卡片等富展示属性。
- 基于笔记本颜色或图标的筛选、搜索、排序能力。

本轮范围严格限定为“创建笔记本时选择图标和颜色，并稳定显示出来”。

## 4. 方案对比与结论

### 方案 A：知识库级集中元数据文件

在知识库下引入一个隐藏元数据文件，统一记录所有顶级笔记本的展示属性，按笔记本路径索引。

优点：

- 读写集中，便于缓存与统一校验。
- 不污染每个笔记本目录，不会让文件树扫描逻辑变复杂。
- 后续要扩展排序、描述、封面等属性时，可以继续沿用同一文件。
- 重建树或迁移前端结构时，元数据合并逻辑单一清晰。

缺点：

- 单个笔记本目录脱离知识库单独拷走时，展示属性不会天然随目录一起走。

### 方案 B：每个笔记本目录各自维护元数据文件

优点：

- 笔记本目录单独拷贝时，展示属性更容易一起迁移。

缺点：

- 每个目录都要多一个系统文件，树扫描和忽略规则更复杂。
- 后续扩展多个展示属性时，元数据将分散在多个目录下，不利于统一治理。
- 更容易把“业务笔记树”和“系统元数据文件”混在一起。

### 方案 C：前端本地存储

优点：

- 开发最快。

缺点：

- 与“跟知识库一起走”的要求冲突。
- 换机器、重开库或共享知识库后展示不一致。
- 必然返工。

### 结论

本轮采用方案 A。

即：使用知识库级集中元数据文件，按顶级笔记本路径统一保存图标与颜色；当前阶段只在创建笔记本时写入，不提供后续编辑入口。

## 5. 数据模型

### 5.1 存储位置

在知识库内新增一个隐藏元数据文件，用于保存笔记本展示配置。

设计要求：

- 该文件不应在普通笔记树中作为用户内容暴露。
- 它属于知识库级配置，而不是某一篇笔记或某一个目录的内容文件。
- 文件名应采用隐藏命名，避免干扰用户浏览。

本设计不强制绑定最终文件名，但要求位置与命名满足上述约束。

### 5.2 记录单位

记录单位是“顶级笔记本路径”。

示例：

- `notes/法律`
- `notes/项目报告`
- `notes/读书`

不记录以下对象：

- `notes` 根目录本身
- `notes/__unarchived__`
- 笔记本内部子目录
- 普通笔记文件

### 5.3 字段模型

建议记录结构如下：

```ts
interface NotebookVisualMetadata {
  notebookPath: string;
  icon: string;
  color: string;
}
```

知识库级文件可组织为路径映射或数组，但语义必须等价：

- `notebookPath`：顶级笔记本路径，作为唯一关联键。
- `icon`：预设图标标识，不直接存渲染组件。
- `color`：预设颜色标识，不直接存任意自由色值。

### 5.4 可扩展性要求

虽然本轮只使用 `icon` 和 `color`，但文件结构必须允许未来扩展以下字段而不破坏兼容性：

- `description`
- `sort_order`
- `cover`
- `updated_at`

因此应避免把当前格式做成只适用于两字段的扁平临时结构。

## 6. 创建交互设计

### 6.1 入口形式

保留当前左侧文件树头部的“新建笔记本”入口，但其点击后不再只显示单个名称输入框，而是展开为一个轻量创建面板。

该面板仍然在左侧栏内完成，不升级为全屏弹窗或独立模态。

### 6.2 面板结构

创建面板从上到下包含：

- 笔记本名称输入框
- 图标预设选择区
- 颜色预设选择区
- 底部操作区：创建、取消

视觉原则：

- 面板应明显比当前单行输入更完整，但仍保持轻量。
- 图标与颜色选择必须一眼可见，不应藏在二级折叠中。
- 预设项数量控制在“小而够用”的范围，不把新建流程变成配置面板。

### 6.3 默认值策略

图标和颜色都必须有默认选中项。

结果：

- 用户只输入名称即可直接创建。
- 不会出现“因为没选图标/颜色而无法提交”的多余阻塞。

### 6.4 预设范围

建议本轮预设控制在以下量级：

- 图标约 8 个。
- 颜色约 8 个。

图标方向建议围绕“知识整理容器”语义，而不是操作图标，例如：文件夹、书本、灯泡、代码、清单、档案、星标、标签。

颜色方向建议使用统一调性的命名预设，而非任意色值输入，例如：蓝、青、绿、橙、红、粉、棕、灰。

### 6.5 提交规则

交互规则如下：

- 名称为空时不能创建。
- Enter 触发创建。
- Esc 取消创建并关闭面板。
- 点击取消关闭面板，不保留本次输入。
- 点击创建后，调用后端统一完成目录创建和元数据写入。

### 6.6 创建成功后的反馈

创建成功后：

- 关闭创建面板。
- 刷新树数据。
- 新笔记本立即以所选图标和颜色显示。

不要求本轮额外做 toast 或全局成功提示，只要侧边栏可立即看到结果即可。

## 7. 侧边栏渲染规则

### 7.1 生效范围

只有顶级笔记本节点应用图标和颜色元数据。

不参与本轮样式映射的节点：

- 普通笔记文件
- 笔记本内部子目录
- `未归档` 分组
- 系统根节点或技术性过渡节点

### 7.2 渲染方式

顶级笔记本在侧边栏中应体现两层视觉信息：

- 图标：强化类型识别。
- 颜色：强化分组记忆。

颜色应作用于节点中最能表达笔记本身份的元素，例如图标本身、节点前导标记或轻量强调色，而不是把整行背景做成高饱和色块。

目标是“有风格但不抢阅读焦点”。

### 7.3 默认回退

当某个顶级笔记本没有元数据时：

- 使用默认图标。
- 使用默认中性色。

默认回退必须稳定且一致，不能随机。

## 8. 读写与回退策略

### 8.1 读路径

树节点渲染时，应在现有笔记树数据基础上叠加笔记本元数据，而不是替换现有树模型。

职责划分建议如下：

- 笔记树仍然负责目录与文件结构。
- 笔记本元数据负责顶级目录的展示属性。
- 前端最终消费的是“结构数据 + 展示属性”的组合结果。

### 8.2 写路径

创建笔记本时，后端必须保证以下两个动作属于同一条产品链路：

- 创建 `notes/<notebook>` 顶级目录。
- 将图标和颜色写入知识库级元数据文件。

前端不应在目录创建成功后再自行做本地补写，否则会造成语义分裂。

### 8.3 异常处理

若元数据文件不存在：

- 按空配置处理，使用默认样式。

若元数据文件损坏、字段非法或出现未知值：

- 忽略非法记录。
- 其余合法记录继续生效。
- 整体树渲染不失败。

若元数据里有目录记录，但目录已不存在：

- 读取时自动忽略。
- 可以在后续写入时顺带清理，但本轮不要求额外提供“清理任务”入口。

### 8.4 一致性原则

展示元数据是“锦上添花”能力，不得影响主数据的可读写。

因此必须满足：

- 没有元数据时，知识库仍然可以正常打开与使用。
- 元数据损坏时，笔记树仍然可以正常显示。
- 元数据丢失时，最多退回默认样式，不能导致笔记本不可见或不可创建。

## 9. 测试策略

### 9.1 前端测试

前端至少覆盖以下内容：

- 新建笔记本面板渲染名称输入、图标预设、颜色预设。
- 使用默认图标与默认颜色创建。
- 切换图标和颜色后，创建接口收到正确参数。
- Esc 与取消动作关闭面板。
- 创建成功后，顶级笔记本节点按元数据显示样式。
- 元数据缺失时回退默认图标与默认颜色。

### 9.2 后端测试

后端至少覆盖以下内容：

- 新建笔记本时同时创建目录并写入元数据。
- 元数据文件不存在时按空配置回退。
- 元数据包含非法字段或未知值时的忽略策略。
- 元数据中存在已删除笔记本记录时的忽略策略。

### 9.3 回归范围

需要额外确认本轮改动不会破坏以下既有行为：

- 现有“新建笔记本”基础创建流程。
- 现有“新建笔记”选择目标笔记本流程。
- 现有 `未归档` 分组展示。
- 现有文件树顶层目录识别规则。

## 10. 风险与后续扩展

### 10.1 风险

- 如果图标标识直接与具体前端组件强耦合，后续替换图标集会放大迁移成本。
- 如果颜色值直接存任意十六进制字符串，后续做统一主题与回退会变复杂。
- 如果把元数据文件暴露进普通树扫描，容易把系统文件误当成用户内容展示。

### 10.2 后续扩展方向

本轮模型完成后，后续可以在同一元数据体系内继续扩展：

- 编辑笔记本图标与颜色。
- 笔记本描述。
- 笔记本排序。
- 笔记本封面或标识图。
- 基于笔记本元数据的视图筛选与个性化展示。

这些扩展不应改变本轮“顶级笔记本路径是唯一关联键”的基本结构。
