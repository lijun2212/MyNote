## 1. 背景与问题定义

当前文件树里的“删除笔记本”确认态采用一条内联浅红提示条，但右侧“删除 / 取消”是偏大的文字按钮。实际效果是：

- 两个动作按钮的视觉重量明显高于提示文本。
- 按钮尺寸与整条确认区的高度不协调，看起来像把小弹框塞进了列表行内。
- 与同一行已有的上移、下移、删除等工具按钮风格不一致。

这次改动目标不是重做交互流程，而是在保留当前内联确认机制的前提下，把操作区压缩成更贴近文件树工具栏语义的紧凑形态。

## 2. 设计目标

- 保留“确认删除该笔记本？”这句提示文字，继续作为主信息。
- 将确认区右侧动作改成紧凑尺寸，避免按钮撑大整条确认区。
- 保留删除动作的危险语义，但不再使用大面积主按钮样式。
- 让确认区的视觉语言与文件树现有工具按钮更一致。
- 不改变当前删除逻辑、错误提示逻辑和确认流程。

## 3. 非目标

- 不修改删除笔记、删除标签等其他删除交互。
- 不引入新的二次确认弹窗或 toast。
- 不修改确认文案与失败文案。
- 不在本次改动中重做整套文件树按钮系统。

## 4. 方案对比

### 4.1 方案 A：纯图标紧凑按钮

做法：右侧改成红色垃圾桶图标和灰色关闭图标，仅保留提示文字。

优点：

- 最紧凑。
- 与工具按钮形态一致。

缺点：

- 删除和取消语义过度依赖图标识别。
- 对高风险动作不够直白。

结论：不采用。

### 4.2 方案 B：图标加极短标签的紧凑按钮（推荐）

做法：右侧改成两个紧凑按钮，分别为“垃圾桶图标 + 删”和“关闭图标 + 取消”。删除按钮保留淡红底或淡红边，取消按钮保持中性灰。

优点：

- 比当前大按钮明显更轻。
- 比纯图标更清晰，删除风险感仍然明确。
- 与文件树现有工具按钮体系更容易对齐。

缺点：

- 相比纯图标仍有少量文字宽度。

结论：采用。

### 4.3 方案 C：纯文字超小按钮

做法：仅缩小“删除 / 取消”的文字按钮尺寸，不引入图标。

优点：

- 代码改动最小。

缺点：

- 视觉问题只被弱化，没有被真正修复。
- 仍然延续当前不协调的按钮语言。

结论：不采用。

## 5. 设计结论

采用方案 B：保留提示文字，右侧改为紧凑图标按钮，并让删除按钮带淡红背景或淡红边框，取消按钮为中性灰。

## 6. 交互与视觉设计

### 6.1 布局

- 整个确认区继续使用当前内联条形结构，不改为浮层。
- 左侧为提示文案“确认删除该笔记本？”。
- 右侧为两个紧凑按钮，水平排列。
- 按钮间距控制在 4px 到 6px。
- 确认区整体内边距比当前版本略收紧，避免形成“按钮比容器更重”的观感。

### 6.2 删除按钮

- 使用垃圾桶图标。
- 保留一个极短标签“删”。
- 默认态使用淡红底或淡红描边，文字与图标使用深红色。
- hover / focus 时允许轻度增强底色或边框，但不应出现当前大主按钮那种强压迫感。

### 6.3 取消按钮

- 使用关闭图标。
- 保留文字“取消”。
- 采用中性灰边框或浅灰底。
- 视觉权重必须低于删除按钮。

### 6.4 尺寸与对齐

- 两个按钮高度应与当前文件树工具按钮处于同一量级，约 18px 到 20px。
- 图标尺寸应小于或等于标签高度，不应出现图标过大导致按钮再次膨胀。
- 行内垂直对齐以提示条高度为准，不能出现按钮上浮或下沉。

## 7. 实现约束

- 仅修改笔记本删除确认条对应的前端组件和测试。
- 尽量复用当前文件树按钮的圆角、边框、hover、focus 语言，避免引入另一套视觉 token。
- 现有 aria-label 与键盘可访问性需保留。
- 删除失败时，确认区继续停留并展示错误信息；本次只调整动作外观，不改变错误处理。

## 8. 测试与验收

### 8.1 测试范围

- 更新现有文件树删除确认测试，使其覆盖新的按钮可点击性与可访问名称。
- 保证删除成功时确认区关闭。
- 保证删除失败时确认区保留并继续显示错误信息。

### 8.2 验收标准

- 删除确认区右侧不再出现视觉过重的大按钮。
- 删除与取消动作明显小于当前版本，并与文件树工具按钮风格统一。
- 保留提示文字，用户不需要依赖纯图标理解动作含义。
- 原有删除成功 / 失败行为不回归。
