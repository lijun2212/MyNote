## 1. 目标与范围

目标：把“总结陈词”升级为“回看摘要”，只服务回看效率，不增加写作负担。

本期范围：

- 在文章开头（标题下方）展示回看摘要。
- 提供半自动候选摘要和轻量编辑能力。
- 维持 `summary` Front Matter 字段作为权威存储键，避免额外迁移成本。
- 将回看摘要纳入搜索和笔记列表摘要展示。

不在本期范围：

- AI 长文本重写。
- 强制每篇填写。
- 批量运营式摘要管理后台。

## 2. 产品约束

必须满足以下约束：

- 默认不强制填写。
- 候选摘要优先，手写摘要兜底。
- 仅高价值笔记触发提示：例如字数阈值、访问频次阈值、被引用次数阈值。
- 日记/流水记录默认不打扰。
- 开头第一屏可见，不能埋在二级面板里。

## 3. 里程碑与排期

- M1（3 天）：数据与接口打通，支持读取/写入/搜索 `summary`。
- M2（4 天）：文章开头回看摘要区块上线，含候选摘要与轻量编辑。
- M3（3 天）：高价值笔记触发策略、提示节流和埋点验证。
- M4（2 天）：回归测试、性能验证和文档收口。

## 4. 任务拆解

### Task 1：语义与命名收口

- 前端文案将“总结陈词”统一替换为“回看摘要”。
- 保持数据字段仍为 `summary`，仅做产品语义升级。
- 更新设计文档和用户提示文案。

交付：

- 文案统一且不出现“强制填写”表达。

### Task 2：开头区块实现

- 在预览区和编辑区顶部增加回看摘要区块（标题下方、正文前）。
- 区块状态：
  - 有摘要：直接展示，支持一键编辑。
  - 无摘要：展示“生成候选摘要”入口和简短提示。
- 区块可折叠但默认展开。

交付：

- 用户打开笔记第一眼可见回看摘要。

### Task 3：半自动候选摘要

- 候选摘要生成采用规则策略（首段 + 一级/二级标题聚合 + 关键词句），先不引入外部模型依赖。
- 候选摘要不自动落库，用户确认后写入 `summary`。
- 若用户已有手写摘要，候选仅做“替换建议”。

交付：

- 点击“生成候选摘要”后可得到可编辑文本。

### Task 4：低负担触发策略

- 仅在满足高价值条件时提示（阈值可配置）：
  - 字数 > N。
  - 最近 7 天访问次数 > M。
  - 反向链接数量 > K。
- 提示节流：同一笔记 24 小时内最多一次提示。
- 日记目录默认关闭提示（可手动打开）。

交付：

- 常规笔记编辑流不被频繁打断。

### Task 5：搜索与列表联动

- 搜索结果中展示回看摘要片段。
- 笔记列表 hover 或详情卡中显示回看摘要。
- 无摘要时保持当前行为，不新增空占位噪音。

交付：

- 回看摘要直接提升检索判断效率。

### Task 6：测试与回归

- 前端测试：
  - 开头区块可见性。
  - 候选摘要生成与确认流程。
  - 提示节流和目录豁免。
- 后端测试：
  - `summary` 写入、读取和搜索命中。
- 端到端测试：
  - 从生成候选到保存摘要再到搜索命中的完整链路。

交付：

- 关键交互与数据链路有自动化覆盖。

## 5. 验收标准

功能验收：

- 不强制填写。
- 用户可在 2 次点击内完成“候选摘要 -> 保存”。
- 开头区块在桌面常规分辨率第一屏可见。
- 搜索可命中回看摘要。

效果验收：

- 回看场景中，找到正确笔记平均耗时下降（相对当前基线）。
- 每日写作流程平均耗时不显著上升。

## 6. 风险与回退

主要风险：

- 候选摘要质量不稳，可能引发“看起来像负担”的主观反感。
- 提示策略阈值不合理，导致过度打扰。

回退策略：

- 提供全局开关：关闭候选与提示，仅保留手写摘要。
- 若质量不达标，降级为“纯手写回看摘要”，保留开头展示价值。
