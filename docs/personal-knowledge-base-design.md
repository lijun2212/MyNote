# 个人 Markdown 笔记知识库方案设计

## 1. 背景与目标

本方案设计一款面向个人长期知识沉淀的跨平台笔记知识库软件。产品体验应接近 Typora：界面简洁、编辑顺滑、以 Markdown 文件作为最终知识载体，同时使用本地 SQLite 数据库管理目录结构、知识关联、总结陈词、知识图谱、修订记录索引与软件设置。

系统优先保证数据可迁移、可离线使用、可长期保存。Markdown 文件是用户知识资产的权威载体，SQLite 是应用层索引与增强能力载体。即使数据库损坏或应用停用，用户仍可直接通过 Markdown 文件访问核心笔记内容。

## 2. 设计原则

1. **文件优先**：Markdown 文件是最终存储格式，不使用私有格式锁定内容。
2. **本地优先**：默认离线可用，所有核心功能不依赖远程服务。
3. **简洁优先**：编辑体验接近 Typora，避免复杂面板长期占用主界面。
4. **索引可重建**：SQLite 中的目录、关联、图谱、摘要等派生数据应尽可能可从文件重建。
5. **跨平台一致**：Windows、macOS、Linux 使用同一套核心逻辑和数据结构。
6. **同步友好**：兼容百度云盘、iCloud、OneDrive、坚果云等基于文件夹同步的软件。
7. **可演进**：知识图谱、自动摘要、AI 辅助等高级能力通过模块化设计逐步加入。

## 3. 用户场景

### 3.1 日常记录

用户打开应用后可以快速新建 Markdown 笔记，使用所见即所得或分屏预览方式编辑内容。文档保存为普通 `.md` 文件，并由系统自动更新索引。

### 3.2 知识管理

用户可以通过目录树、标签、双链、反向链接、搜索和知识图谱查找已有内容。系统根据 Markdown 标题、链接、标签、引用关系和用户手动关联生成知识关系网络。

### 3.3 总结沉淀

每篇笔记可维护一段“总结陈词”，用于表达这篇笔记的核心观点、结论或当前理解。总结既可人工编辑，也可在后续版本中接入本地或远程 AI 辅助生成。

### 3.4 修订追踪

用户可以查看笔记的历史变更记录，包括修改时间、标题变化、摘要变化、关键内容片段变化和文件版本快照。修订记录用于个人回溯，不替代 Git 等专业版本管理工具。

### 3.5 多设备同步

用户将知识库根目录放在百度云盘等公有云同步目录中。应用通过文件监听、冲突检测和索引重建机制适配同步造成的文件变更。

## 4. 功能设计

### 4.1 知识库管理

- 创建、打开、关闭知识库。
- 一个知识库对应一个本地文件夹。
- 文件夹内包含 Markdown 文件、附件目录和应用元数据目录。
- 支持最近打开知识库列表。
- 支持知识库级别设置，例如默认编辑模式、主题、同步策略、附件路径。

推荐目录结构：

```text
MyKnowledgeBase/
  notes/
    work/
      meeting.md
    reading/
      book-a.md
  assets/
    images/
    files/
  .mynote/
    index.sqlite
    backups/
    logs/
    config.json
```

### 4.2 Markdown 编辑

- 支持标准 Markdown 语法。
- 支持标题、列表、引用、代码块、表格、任务列表、脚注、数学公式、图片。
- 支持实时预览。
- 支持三种编辑视图：
  - 纯编辑模式。
  - 分屏预览模式。
  - 类 Typora 的所见即所得模式。
- 支持自动保存。
- 支持快捷键、查找替换、字数统计、文档大纲。
- 支持粘贴图片后自动保存到 `assets/images/` 并插入相对路径。

### 4.3 文件与目录管理

- 左侧目录树展示 Markdown 文件与文件夹。
- 支持新建、重命名、移动、删除文件和文件夹。
- 文件操作必须同步更新 SQLite 索引。
- 支持从外部文件管理器修改后自动识别变更。
- 支持软删除：删除时先移入知识库回收站，避免误删。

### 4.4 标签与分类

- 支持在 Markdown Front Matter 中声明标签：

```yaml
---
title: 示例笔记
tags: [阅读, 软件工程]
summary: 这是一篇示例笔记的核心结论。
---
```

- 支持从正文中识别 `#标签`。
- 支持按标签过滤笔记。
- 支持标签重命名、合并和删除。

### 4.5 双链与反向链接

- 支持 Wiki 链接语法：`[[笔记标题]]`、`[[笔记标题#章节]]`、`[[笔记标题|显示文本]]`。
- 支持普通 Markdown 链接解析。
- 笔记详情页展示：
  - 当前笔记链接到哪些笔记。
  - 哪些笔记引用了当前笔记。
  - 未解析链接列表。
- 当目标笔记不存在时，可一键创建。

### 4.6 知识关联管理

除自动解析链接外，系统支持用户手动建立关系：

- 相关。
- 前置知识。
- 延伸阅读。
- 反驳。
- 支撑。
- 同义或相似。

每条关系可包含说明文字、创建时间和来源。

### 4.7 总结陈词

每篇笔记支持一个结构化总结字段：

- 优先存储在 Markdown Front Matter 的 `summary` 字段中。
- SQLite 保存冗余索引，便于搜索和列表展示。
- 支持在笔记详情侧栏展示。
- 支持历史记录，记录总结内容的变化。

### 4.8 知识图谱

知识图谱基于以下关系生成：

- Wiki 链接。
- Markdown 链接。
- 标签共现。
- 手动关系。
- 同目录相邻关系。

图谱功能包括：

- 当前笔记局部图谱。
- 全库知识图谱。
- 按标签、目录、关系类型筛选。
- 点击节点打开笔记。
- 高亮孤立节点、中心节点和未解析链接。

首版建议优先实现“当前笔记的一跳/两跳局部图谱”，全库复杂布局可放到后续版本优化。

### 4.9 搜索

- 支持标题搜索。
- 支持正文全文搜索。
- 支持标签搜索。
- 支持路径搜索。
- 支持总结搜索。
- 支持链接关系搜索。

SQLite 使用 FTS5 建立全文索引。文件变更后异步更新索引。

### 4.10 修订记录

修订记录分为两层：

1. **轻量变更日志**：记录文件路径、修改时间、内容哈希、标题、摘要、字数变化。
2. **可选快照**：按策略保存 Markdown 内容快照，例如每 10 分钟一次、每天第一次修改一次、手动保存重要版本。

修订记录用于应用内查看差异和回滚。考虑到云盘同步可能引入外部修改，系统需要根据文件哈希识别变化来源。

### 4.11 同步支持

应用本身不直接实现百度云盘协议，而是兼容公有云同步文件夹：

- 用户将知识库目录放入百度云盘同步目录。
- 应用监听本地文件变化。
- 检测到远程同步写入后，重新解析变更文件。
- SQLite 索引可重建，避免同步数据库冲突造成知识丢失。

推荐同步策略：

- Markdown 文件、附件和配置文件参与云盘同步。
- SQLite 索引默认可以同步，但必须支持冲突恢复。
- 若发现多个数据库冲突副本，以 Markdown 文件为准重建索引。

### 4.12 设置

设置分为应用级和知识库级：

- 应用级：主题、字体、界面语言、快捷键、最近打开记录。
- 知识库级：附件目录、默认保存路径、图谱深度、自动保存间隔、修订策略、同步策略。

## 5. 架构设计

### 5.1 推荐技术路线

推荐采用 **Tauri + Web 前端 + Rust 核心服务 + SQLite**。

理由：

- 跨平台能力强，支持 Windows、macOS、Linux。
- 比 Electron 更轻量，安装包和内存占用更低。
- Rust 适合文件系统监听、SQLite 访问、索引构建等本地能力。
- 前端可使用成熟 Markdown 编辑器、图谱渲染和 UI 生态。

备选路线：

| 路线 | 优点 | 缺点 | 适用情况 |
| --- | --- | --- | --- |
| Tauri + Rust + Web | 轻量、跨平台、系统集成好 | Rust 学习和工程复杂度较高 | 推荐路线 |
| Electron + Node.js + Web | 生态成熟、开发快 | 体积大、资源占用高 | 快速验证产品 |
| Flutter Desktop | 跨平台 UI 一致 | Markdown 编辑器和 Web 图谱生态较弱 | 强调原生 UI 一致性 |

### 5.2 分层架构

```text
┌─────────────────────────────────────────────┐
│ UI 层                                        │
│ 编辑器 / 预览 / 目录树 / 搜索 / 图谱 / 设置   │
├─────────────────────────────────────────────┤
│ 应用服务层                                   │
│ 笔记服务 / 搜索服务 / 图谱服务 / 修订服务      │
├─────────────────────────────────────────────┤
│ 核心领域层                                   │
│ Note / Tag / Link / Relation / Revision      │
├─────────────────────────────────────────────┤
│ 基础设施层                                   │
│ 文件系统 / SQLite / FTS5 / 文件监听 / 配置     │
└─────────────────────────────────────────────┘
```

### 5.3 核心模块

#### 5.3.1 UI 模块

负责用户交互和状态展示：

- 主窗口布局。
- Markdown 编辑器。
- 实时预览。
- 文件目录树。
- 反向链接面板。
- 图谱视图。
- 搜索面板。
- 修订记录面板。
- 设置页。

#### 5.3.2 笔记服务

负责 Markdown 文件的生命周期：

- 创建笔记。
- 读取笔记。
- 保存笔记。
- 重命名和移动笔记。
- 删除和恢复笔记。
- 解析 Front Matter。
- 提取标题、标签、摘要、链接和正文纯文本。

#### 5.3.3 索引服务

负责将文件内容写入 SQLite：

- 文件路径索引。
- 标题索引。
- 标签索引。
- 链接索引。
- 全文搜索索引。
- 内容哈希。
- 字数和更新时间。

索引服务必须支持全量重建和增量更新。

#### 5.3.4 图谱服务

负责从链接、标签和手动关系生成图数据：

- 查询当前笔记的一跳关系。
- 查询当前笔记的两跳关系。
- 查询全库关系。
- 合并重复边。
- 输出前端图谱渲染所需的节点和边。

#### 5.3.5 修订服务

负责记录和展示变化：

- 保存修订元数据。
- 按策略保存内容快照。
- 计算版本差异。
- 支持恢复到指定版本。

#### 5.3.6 同步适配服务

不直接连接公有云 API，而是处理云盘同步后的本地文件变化：

- 文件监听。
- 防抖处理。
- 冲突文件识别。
- 数据库冲突检测。
- 索引重建提示。

## 6. 数据设计

### 6.1 Markdown Front Matter

每篇笔记建议包含以下元数据：

```yaml
---
id: 01HZXAMPLE000000000000000000
title: 笔记标题
created_at: 2026-05-29T16:00:00+08:00
updated_at: 2026-05-29T16:10:00+08:00
tags: [示例, 知识库]
summary: 这篇笔记的核心结论。
---
```

其中 `id` 是稳定标识，用于避免文件重命名后关系丢失。若用户从外部添加 Markdown 文件，系统应在首次索引时补齐缺失的 `id`，或在数据库中维护路径到临时 ID 的映射。

### 6.2 SQLite 表设计

#### knowledge_bases

记录知识库信息。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | TEXT PRIMARY KEY | 知识库 ID |
| name | TEXT | 名称 |
| root_path | TEXT | 根目录 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

#### notes

记录笔记索引。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | TEXT PRIMARY KEY | 笔记稳定 ID |
| path | TEXT UNIQUE | 相对路径 |
| title | TEXT | 标题 |
| summary | TEXT | 总结陈词 |
| content_hash | TEXT | 内容哈希 |
| word_count | INTEGER | 字数 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |
| indexed_at | TEXT | 索引时间 |
| deleted_at | TEXT | 软删除时间 |

#### tags

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | TEXT PRIMARY KEY | 标签 ID |
| name | TEXT UNIQUE | 标签名 |
| normalized_name | TEXT UNIQUE | 归一化标签名 |

#### note_tags

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| note_id | TEXT | 笔记 ID |
| tag_id | TEXT | 标签 ID |

#### links

记录自动解析出的链接。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | TEXT PRIMARY KEY | 链接 ID |
| source_note_id | TEXT | 来源笔记 |
| target_note_id | TEXT | 目标笔记，可为空 |
| target_raw | TEXT | 原始链接文本 |
| link_type | TEXT | wiki / markdown / heading |
| anchor | TEXT | 章节锚点 |
| resolved | INTEGER | 是否已解析 |

#### relations

记录用户手动建立的知识关系。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | TEXT PRIMARY KEY | 关系 ID |
| source_note_id | TEXT | 来源笔记 |
| target_note_id | TEXT | 目标笔记 |
| relation_type | TEXT | 关系类型 |
| description | TEXT | 关系说明 |
| created_at | TEXT | 创建时间 |

#### revisions

记录修订元数据。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | TEXT PRIMARY KEY | 修订 ID |
| note_id | TEXT | 笔记 ID |
| path | TEXT | 当时路径 |
| content_hash | TEXT | 内容哈希 |
| title | TEXT | 当时标题 |
| summary | TEXT | 当时总结 |
| word_count | INTEGER | 当时字数 |
| snapshot_path | TEXT | 快照路径，可为空 |
| created_at | TEXT | 创建时间 |

#### app_settings / kb_settings

使用 key-value 结构保存设置：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| key | TEXT PRIMARY KEY | 设置键 |
| value | TEXT | JSON 字符串 |
| updated_at | TEXT | 更新时间 |

#### note_fts

使用 SQLite FTS5：

```sql
CREATE VIRTUAL TABLE note_fts USING fts5(
  note_id UNINDEXED,
  title,
  summary,
  body
);
```

## 7. 关键流程设计

### 7.1 创建笔记

1. 用户在目录树中选择位置并创建笔记。
2. 应用生成稳定 `id`。
3. 写入包含 Front Matter 的 Markdown 文件。
4. 解析文件并写入 `notes`、`note_fts`、`tags` 等表。
5. UI 刷新目录树、编辑器和图谱。

### 7.2 保存笔记

1. 编辑器内容变更后触发防抖自动保存。
2. 写入 Markdown 文件。
3. 计算内容哈希。
4. 若哈希变化，记录修订元数据。
5. 异步更新索引、链接、标签和全文搜索。
6. 通知 UI 更新反向链接和图谱。

### 7.3 外部文件变更

1. 文件监听器发现 Markdown 文件变化。
2. 等待短暂防抖，避免读取同步中的半成品文件。
3. 读取文件并计算哈希。
4. 若文件新增，则创建索引记录。
5. 若文件修改，则更新索引并记录修订。
6. 若文件删除，则标记为删除或从索引移除。

### 7.4 链接解析

1. Markdown 解析器提取 Wiki 链接和普通 Markdown 链接。
2. 根据笔记 ID、标题、别名、路径尝试解析目标。
3. 解析成功则写入 `target_note_id`。
4. 解析失败则保留 `target_raw`，在 UI 中展示为未解析链接。
5. 当新笔记创建或标题变化时，重新解析相关链接。

### 7.5 图谱生成

1. 查询当前笔记的链接关系和手动关系。
2. 根据配置扩展一跳或两跳节点。
3. 合并相同来源和目标的边。
4. 给节点附加标题、路径、标签、摘要。
5. 前端使用图谱组件渲染。

### 7.6 索引重建

1. 用户手动触发或系统检测到数据库异常。
2. 扫描知识库下所有 Markdown 文件。
3. 解析 Front Matter、正文、标签、链接。
4. 重建 `notes`、`tags`、`links`、`note_fts` 等可派生表。
5. 保留不可派生数据，例如用户手动关系、设置和修订记录；若数据库不可用，则从备份恢复。

## 8. UI 设计

### 8.1 主界面布局

```text
┌────────────────────────────────────────────────────────────┐
│ 顶栏：知识库名 / 快速搜索 / 同步状态 / 设置                 │
├───────────────┬────────────────────────────┬───────────────┤
│ 目录与标签     │ Markdown 编辑与实时预览      │ 关联信息        │
│               │                            │               │
│ - 文件树       │ - 编辑模式                  │ - 大纲          │
│ - 标签         │ - 分屏预览                  │ - 反向链接      │
│ - 最近笔记     │ - 所见即所得                │ - 总结陈词      │
│               │                            │ - 修订记录      │
├───────────────┴────────────────────────────┴───────────────┤
│ 底栏：字数 / 保存状态 / 当前路径 / 索引状态                  │
└────────────────────────────────────────────────────────────┘
```

### 8.2 简洁模式

为了接近 Typora，应提供“专注写作模式”：

- 隐藏左右侧栏。
- 仅显示编辑区域。
- 顶栏和底栏弱化。
- 鼠标移至边缘时再显示目录和关联面板。

### 8.3 知识关系面板

右侧面板按优先级展示：

1. 当前文档大纲。
2. 总结陈词。
3. 出链。
4. 反向链接。
5. 相关笔记。
6. 修订记录。

### 8.4 图谱视图

图谱作为独立视图或弹出面板，不长期占据编辑区。用户从当前笔记进入图谱时，默认展示当前笔记的局部网络，避免全库图谱过于复杂。

## 9. 跨平台与本地能力

### 9.1 文件路径

- 数据库存储相对路径。
- UI 展示平台原生路径格式。
- 内部统一使用规范化路径。
- 避免在 Markdown 中写入绝对附件路径。

### 9.2 文件监听

不同平台文件监听行为不同，需要统一封装：

- macOS：FSEvents。
- Windows：ReadDirectoryChangesW。
- Linux：inotify。

Tauri/Rust 侧可使用成熟跨平台监听库封装差异。

### 9.3 打包发布

- macOS：`.dmg`。
- Windows：`.msi` 或 `.exe`。
- Linux：`.AppImage` 或 `.deb`。

## 10. 同步与冲突策略

### 10.1 推荐同步模式

用户自行安装百度云盘等同步客户端，并将知识库文件夹放入同步目录。应用只负责保证本地文件结构和索引恢复能力。

### 10.2 冲突类型

| 类型 | 例子 | 处理方式 |
| --- | --- | --- |
| Markdown 文件冲突 | `note.md` 和 `note(冲突副本).md` | 保留两份，提示用户合并 |
| SQLite 冲突 | `index.sqlite` 出现冲突副本 | 以 Markdown 为准重建索引 |
| 附件冲突 | 图片同名冲突 | 重命名保留，更新引用需用户确认 |
| 设置冲突 | config 出现多版本 | 使用最近修改版本并保留备份 |

### 10.3 数据安全

- 保存前写入临时文件，再原子替换。
- 定期备份 SQLite。
- 修订快照限制总大小，避免无限增长。
- 索引重建前备份旧数据库。

## 11. 错误处理设计

- Markdown 文件读取失败：在目录树中标记异常，并展示错误原因。
- SQLite 打开失败：提示用户从备份恢复或根据 Markdown 重建索引。
- 链接解析失败：不阻断编辑，展示为未解析链接。
- 文件同步中：延迟索引，避免读取未完成文件。
- 自动保存失败：明确提示用户，并保留编辑器内未保存内容。
- 快照写入失败：不影响主文件保存，但需要提示修订记录不完整。

## 15. 首版范围建议

首版应聚焦“可靠写作 + 本地知识组织”，避免一次性做过多高级能力。

建议首版包含：

- 知识库创建和打开。
- Markdown 编辑和实时预览。
- 文件目录树。
- SQLite 索引。
- 标签。

## 16. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 云盘同步导致数据库冲突 | 索引异常 | Markdown 为准，支持重建索引 |
| 所见即所得编辑器复杂 | 开发周期变长 | 首版先支持编辑+预览，再增强 Typora 模式 |
| 知识图谱全库渲染卡顿 | 体验差 | 首版默认局部图谱 |
| 文件与数据库状态不一致 | 搜索和关联错误 | 文件监听、哈希校验、索引重建 |
| 修订快照占用过大 | 磁盘膨胀 | 配置保留策略和容量上限 |
| 跨平台路径差异 | 文件引用失效 | 内部统一相对路径和路径规范化 |

## 17. 推荐结论

推荐采用 **Tauri + Rust + Web 前端 + SQLite + Markdown 文件存储** 的本地优先架构。Markdown 文件作为最终知识资产，SQLite 作为可重建索引和增强数据层。首版优先完成简洁编辑、实时预览、目录管理、标签、双链、搜索、总结和局部图谱；同步能力通过兼容公有云同步文件夹实现，而不是直接接入云盘 API。

该方案能够兼顾 Typora 式简洁体验、知识库长期可维护性、跨平台使用和未来扩展能力。
